\chapter{Preliminaries for Single-Robot Temporal Task Planning}
\label{chap:preliminaries}

Automated motion and task planning framework for a mobile robot is introduced in this chapter as the nominal solution.
In particular, a finite-state transition system is constructed to serve as the discrete abstraction of the  robot motion within the workspace.
The robot is assigned a complex task which is specified as temporal logic formulas over this transition system.
Through the model-checking algorithm, we synthesize firstly a discrete motion plan that satisfies the given task and more importantly minimizes a total cost. Then based on this discrete plan, we construct a hybrid control strategy that drives the robot such that its final trajectory satisfies the task specification.

\section{Abstraction of Robot Motion}
\label{ch02:sec:system_abstraction_task_specification}
The robot's  motion within a certain workspace is abstracted as a finite-state transition system (FTS)~\citep{baier2008principles}. This FTS is constructed by integrating two aspects: (i) the workspace model; (ii) the robot dynamics and its navigation controller.
%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%
\subsection{The workspace model}
\label{ch02:subsec:workspacemodel}
The workspace we consider is a bounded $n$-dimensional space, denoted by $\mathcal{W}_0 \subset \mathbb{R}^n$, within which there exists $W$ smaller regions of interest $\pi_i \subset \mathcal{W}_0$, $\forall i=1,2,\cdots,W$. Denote by $\Pi=\{\pi_1,\cdots,\pi_W\}$ the set of all smaller regions. We require that $\Pi$ is a full partition of the workspace and any two regions~$\pi_i, \pi_j\in \Pi$ do not overlap. Namely,  $\bigcup_{i=1}^W \pi_i = \mathcal{W}_0$ and $\pi_i\cap \pi_j=\emptyset$, $\forall i,j=1,2,\ldots,W$ and $i\neq j$.

Atomic propositions are boolean variables that can be either~\texttt{True} or~\texttt{False}, which are denoted by~$\top$ and~$\bot$ for brevity. They are used  to express known properties about the state of the robot. Specifically, in order to indicate the robot's position, we define the set of atomic propositions $AP_r=\{a_{r,i}\}$,   $i=1,\cdots,W$, where
\begin{equation}\label{ch02:region}
a_{r,i}=\left \{ \begin{array}{c l}
   \top & \quad \text{if the robot is in region $\pi_i$},\\
   \bot & \quad \text{otherwise},
\end{array}\right.
\end{equation}
where $a_{r,i}$ can be evaluated by monitoring the measurements from a localization or positioning system. The requirement that $\bigcup_{i=1}^n \pi_i = \mathcal{W}_0$ is important to ensure that the robot's position is tracked at all time. Beside the geometric structure of the workspace, we also would like to express some generic properties within the workspace that are of interest to potential tasks. Denote by $AP_p=\{a_{p,1},a_{p,2},\cdots, a_{p,M}\}$ the set of atomic propositions for these properties. For simplicity, we set
%\begin{equation}\label{ch02:eq:AP}
$AP=AP_r\cup AP_p$
%\end{equation}
 as the set of all atomic propositions.

\begin{definition}\label{ch02:def:labeling}
The labeling function $L: \Pi \rightarrow 2^{AP}$ maps a region $\pi_i\in \Pi$ to the set of atomic propositions satisfied by $\pi_i$, and $a_{r,i}\in L(\pi_i)$ by default, $\forall i=1,\cdots,W$. \hfill $\blacktriangle$
\end{definition}
Note that partial satisfaction of a proposition is not allowed. Namely, if only a part of region $\pi_i$ satisfies $a\in AP$ and the other part does not, then $\pi_i$ should be split into two regions: one satisfies $a$ and the other does not.  $AP_p$ greatly improves the flexibility when expressing the intended tasks because one generic property can represent one type of regions without explicitly specify the location of these regions.

\begin{example}\label{ch02:examp:partition}
  An office-like workspace has six rooms and one corridor, which gives the partition in Figure~\ref{ch02:fig:partition}. Two properties are ``there is a basket in the region'' and ``there is a ball in the region''.
  \hfill $\blacktriangle$
\end{example}

Additionally, since every region is a dense subset of the $n$-dimensional space, it is impossible to represent each region by the set of points contained in it. Thus it is crucial to represent and encode these regions efficiently. For instance, rectangles can be encoded by its center coordinate, height and width; sphere by its center and radius; triangular by the coordinates of its corners. There are also automated partition tools like Delaunay triangulation~\citep{lee1980two} and Voronoi diagram~\citep{lavalle2006planning}. Note that this level of partition is preliminary and not robot-specific.
%, i.e., it does not depend on agent dynamics and navigation in Section~\ref{ch02:subsec:robotdynamics}.

\subsection{Robot dynamics and navigation controller}
\label{ch02:subsec:robotdynamics}
We assume the robot satisfies the following continuous dynamics:
\begin{equation}\label{ch02:eq:dynamic}
\dot{x}(t)=f(x(t), u(t)),
\end{equation}
where $x\in \mathbb{R}^n$, $u\in \mathbb{R}^m$ are the position and control signal; $f(\cdot)$ is Lipschitz continuous~\citep{khalil2002nonlinear}. Thus the system is deterministic, i.e.,  given an initial state $x(0)$, a control input $u(t):\mathbb{R}\rightarrow \mathbb{R}^m$ produces an unique state trajectory. We mainly take into account the single-integrator dynamics or the unicycle model. However the proposed framework can be potentially applied to other dynamics.

Given the preliminary partition from Section~\ref{ch02:subsec:workspacemodel} and the agent dynamics by~\eqref{ch02:eq:dynamic}, we need to abstract the robot's ability to transit from one region to another, which is not necessarily the adjacency relation in the geometrical sense. Instead it is defined in the control-driven fashion.
%========
\begin{definition}\label{ch02:def:transition}
There is a transition from $\pi_i$ to $\pi_j$ if an admissible navigation controller $\mathcal{U}:\mathbb{R}^n\times \Pi \times \Pi \rightarrow \mathbb{R}^m$:
\begin{equation}\label{ch02:eq:control}
u(t) = \mathcal{U}(x(t), \pi_i,\, \pi_j), \qquad t \in [t', \, t'']
\end{equation}
exists that could drive the system~\eqref{ch02:eq:dynamic} from \textbf{any} point in region $\pi_i$ to a point in region $\pi_j$ in finite time. At the same time the robot should stay within $\pi_i$ or $\pi_j$. Namely, $x(t')\in \pi_i$, $x(t'')\in \pi_j$ and $x(t) \in \pi_i \cup \pi_j$,  $\forall t\in [t', \, t'']$. \hfill $\blacktriangle$
\end{definition}
%==========
The above definition is closely related to the implementation of a discrete motion plan described in Section~\ref{ch02:sec:hybrid}. Two main navigation techniques are discussed in the thesis: (i) \cite{lindemann2006real} proposes a potential-field-based feedback control algorithm. It navigates a differential-driven mobile robot from any point inside a region to an adjacent region through a desired facet. The partition is based on generalized Voronoi diagram and a smooth vector filed is constructed over each triangular region; (ii) \cite{koditschek1990robot} provides a provably correct point-to-point navigation algorithm, by constructing an exact navigation function for sphere workspace and obstacles. It has been successfully applied in both single~\citep{loizou2008density} and multi-agent~\citep{dimarogonas2007decentralized} navigation control under different geometric constraints, like single-integrator~\citep{loizou2002closed}, double-integrator~\citep{dimarogonas2005decentralized} vehicles. By following the negated gradient of the navigation function, a collision free path is guaranteed from almost any initial position in the free space to any goal position in the free space given that the workspace is valid.

%\item Sampling-based motion planning techniques like probabilistic roadmap method~\cite{kavraki1996probabilistic}, and rapidly-exploring random trees~\cite{lavalle1998rapidly},~\cite{karaman2011sampling} have achieved great success both in theory and practice, for various geometrically constrained models~\cite{ladd2005fast}. The required controller for feasible trajectories are constructed automatically as a result of the random exploration process of the configuration space, of which the optimality can be improved by increasing the number of samples.

\subsection{Control-driven and weighted FTS}
\label{ch02:subsec:FTS}
The robot motion is abstracted as transitions among the regions $\Pi=\{\pi_1,\,\cdots, \pi_N\}$. With a slight abuse of notation, we denote by $\pi_i=\{\texttt{the robot is at region}\; \pi_i\}$, $i=1,\cdots,N$, the states that reflect which region  the robot is currently visiting.  The transitions or state changes represent that the robot has moved from one region to another. Recall that the transition relation is not necessarily the adjacency relation in the geometrical sense, but by Definition~\ref{ch02:def:transition}. Formally the control-driven and weighted finite-state transition system (FTS) is defined below:
\begin{definition}\label{ch02:def:FTS}
The weighted FTS is a tuple $\mathcal{T}_c=(\Pi,\, \longrightarrow_c,\, \Pi_0,\,AP,\, L_c,\, W_c)$, where
$\Pi=\{\pi_i, \, i=1,\cdots,W\}$ is the set of states;
$\longrightarrow_c\, \subseteq \Pi \times \Pi$ is the transition relation by Definition~\ref{ch02:def:transition}. For simplicity, $\pi_i \longrightarrow_c \pi_j$ is equivalent to $(\pi_i,\,\pi_j)\in \longrightarrow_c$;
$\Pi_0\subseteq \Pi$ is the initial state, to indicate where the robot may start from;
$AP = AP_r\cup AP_p$ is the set of atomic propositions;
$L_c: \, \Pi\,\rightarrow\, 2^{AP}$ is a labeling function by Definition~\ref{ch02:def:labeling};
$W_c:\, \Pi\times \Pi \,\rightarrow\, \mathbb{R}^+$ is the weight function as the cost of a transition in $\longrightarrow_c$. \hfill $\blacktriangle$
\end{definition}
%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%

We assume that $\mathcal{T}_c$ does not have a terminal state. The {successors} of state $\pi_i$ are defined as $\texttt{Post}(\pi_i)=\{\pi_j \in \Pi \, | \, \pi_j \longrightarrow_c \pi_j \}$. An infinite \emph{path} of $\mathcal{T}_c$ is  an infinite state  sequence $\tau=\pi_1\pi_2 \cdots$ such that $\pi_1 \in \Pi_0$ and $\pi_i\in \texttt{Post}(\pi_{i-1})$ for all $i>0$. The \emph{trace} of a path is the sequence of sets of atomic propositions that are true in the states along that path, i.e., $\texttt{trace}(\tau)=L_c(\pi_1)L_c(\pi_2)\cdots$. The trace of $\mathcal{T}_c$ is defined as $\texttt{Trace}(\mathcal{T}_c)=\cup_{\tau \in I }\, \texttt{trace}(\tau)$, where $I$ is the set of all infinite paths in $\mathcal{T}_c$. An useful way to represent an infinite path is to use the $\omega$-operator, to indicate the segment that has to be repeated infinitely many times~\citep{baier2008principles}.


The weighted FTS $\mathcal{T}_c$ is \emph{fully-known} if it reflects the actual workspace model and robot dynamics; $\mathcal{T}_c$ is called \emph{static} if $\mathcal{T}_c$ does not change with time.

\section{Task Specification as LTL Formulas}\label{ch02:sec:LTL}
%=============================
A language is needed to specify a complex task, which on one hand should be expressive enough to specify various types of tasks, and on the other hand should be formal enough to avoid ambiguity and misinterpretation.  Linear time Temporal Logic (LTL) provides a concise and formal way to specify both propositional and temporal constraints on the system behavior.

%=============================
\subsection{Syntax and semantics}\label{ch02:subsec:syntax}
Linear-time temporal logic (LTL) is defined using the following syntax:
\begin{equation}\label{ch02:eq:LTLsyntax}
\varphi::=\top \;|\; a  \;|\; \varphi_1 \vee \varphi_2  \;|\; \neg \varphi  \;|\; \bigcirc \varphi  \;|\;  \varphi_1\, \textsf{U}\, \varphi_2,
\end{equation}
where $a \in AP$ and $\wedge$ (\emph{or}), $\neg$ (\emph{not}), $\bigcirc$ (\emph{next}), $\textsf{U}$ (\emph{until}). For brevity, we omit the derivations of other useful operators like $\square$ (\emph{always}), $\Diamond$ (\emph{eventually}), $\Rightarrow$ (\emph{implication}) and refer to Chapter~5 of~\cite{baier2008principles}. An infinite \emph{word} over the alphabet $2^{AP}$ is an infinite sequence $\sigma \in (2^{AP})^{\omega}$ that $\sigma = S_0\, S_1\, S_2 \cdots$, where $S_k \in 2^{AP}$ for all $k=1,\,2,\,\cdots$, where $S_k$ is the set of atomic propositions that are true at time step $k$.

The semantics of LTL for an infinite word $\sigma$ is given via a doubly-recursive definition of the relation $(\sigma,\,k)\models \varphi$, i.e., $\sigma$ \emph{satisfies} $\varphi$ at time step $k$.
\begin{definition}\label{ch02:def:semantics}
The semantics of LTL is defined as follows:
\begin{equation*}
\begin{split}
(\sigma,\,k)\models a \quad & \leftrightarrow \quad \, a \in S_k \\
(\sigma,\,k)\models \neg \, \varphi \quad & \leftrightarrow \quad (\sigma,\, k) \nvDash \varphi\\
(\sigma,\,k)\models \bigcirc\, \varphi \quad & \leftrightarrow \quad (\sigma,\, k+1) \models \varphi\\
(\sigma,\,k)\models \varphi_1 \vee \varphi_2 \quad & \leftrightarrow \quad  (\sigma,\,k)\models \varphi_1 \,\, \text{or}\,\,  (\sigma,\,k)\models \varphi_2 \\
(\sigma,\,k)\models \varphi_1\, \textup{\textsf{U}} \, \varphi_2 \quad & \leftrightarrow \quad  \exists k'\in [k,\,+\infty], \, (\sigma,\, k')\models \varphi_2 \,\, \text{and}\\
 &  \quad \quad \,\,\, \forall k''\in (k,\, k'),\, (\sigma,\, k'')\models \varphi_1\,. \qquad \qquad \qquad \qquad \blacktriangle
\end{split}
\end{equation*}
\end{definition}

Any LTL formula $\varphi$ is satisfied by $\sigma$ at time step $0$ if $(\sigma,\, 0)\models \varphi$ (for simplicity we denote by $\sigma \models \varphi$). The \emph{words} of $\varphi$ is defined as the set of words that satisfy $\varphi$ at time step $0$, i.e., $\texttt{Words}(\varphi)=\{\sigma \in (2^{AP})^{\omega}\,|\, \sigma \models \varphi\}$.
Given an infinite path~$\tau$  of $\mathcal{T}_c$ and an LTL formula $\varphi$ over $AP$, $\texttt{trace}(\tau)$ is a word over the alphabet $2^{AP}$. Thus we can verify if $\texttt{trace}(\tau)$ satisfies $\varphi$ according to the semantics~\eqref{ch02:def:semantics}.
\begin{definition}\label{ch02:def:path-satisfy}
An infinite path $\tau$ \textbf{satisfies} $\varphi$, i.e., $\tau\models \varphi$ if its trace $\textup{\texttt{trace}}(\tau)\models \varphi$. A satisfying path is also called a \textbf{plan} for $\varphi$. \hfill $\blacktriangle$
\end{definition}



LTL formulas can be used to specify various robot control tasks, such as safety ($\square \neg\, \varphi_1$, globally avoiding $\varphi_1$), ordering ($\Diamond(\varphi_1\, \wedge \Diamond \, (\varphi_2 \,\wedge \Diamond \,\varphi_3))$, $\varphi_1$, $\varphi_2$, $\varphi_3$ hold in sequence), response ($\varphi_1 \Rightarrow \varphi_2$, if $\varphi_1$ holds, $\varphi_2$ will hold in future), repetitive surveillance ($\square\Diamond \varphi$, $\varphi$ holds infinitely often).

Another particular class of LTL we consider in this thesis  is the syntactically co-safe LTL (sc-LTL)~\cite{kupferman2001model}.
They only contain the $\bigcirc$, $\textsf{U}$ and $\Diamond$ operators and are written in positive normal form~\cite{fainekos2009robustness}.
In contrast, the satisfaction of an sc-LTL formula can be achieved in a finite time, i.e., each word satisfying an sc-LTL formula $\varphi$ consists of a \emph{satisfying prefix} that can be followed by an arbitrary suffix.

As discussed in the introduction, a single counter example would be enough for the purpose of verification, i.e., to verify that not all infinite paths of $\mathcal{T}_c$ satisfy~$\varphi$. However for plan synthesis, since the derived plan needs to be implemented by autonomous robots, we need to find a plan that fulfills certain structure. % and is optimized in terms of certain costs.

%=============================
\subsubsection{Prefix-suffix structure}\label{ch02:subsec:prefix-suffix}
As a plan is essentially an infinite sequence of states in $\mathcal{T}_c$, it is not convenient to encode, analyze or manipulate both in theory and software implementation. Thus we consider the plan with the \emph{prefix-suffix} structure:
\begin{equation}\label{ch02:eq:prefix-suffix}
\tau = \langle \tau_{\text{pre}}, \tau_{\text{suf}}\rangle =
 \tau_{\text{pre}}\, [\tau_{\text{suf}}]^\omega
%\pi_0\, \pi_1\,\cdots \pi_{f-1} [\,\pi_{f}\,\pi_{f+1}\cdots \pi_{n}]^{\omega}\, ,
\end{equation}
% where the  {prefix} part $\tau_{\text{pre}}=(\pi_0\, \pi_1\, \pi_2\cdots \pi_{f-1})$ from an initial state $q_0$ to some state $q_{f-1}$ that is executed only once and the  {suffix} part $\tau_{\text{suf}}=(\,\pi_{f}\,\pi_{f+1}\cdots \pi_{n})$ from $\pi_f$ back to itself that is repeated infinitely. Note that there are no correspondences among the subscripts. A satisfying path with this prefix-suffix structure has a finite representation as~\eqref{ch02:eq:prefix-suffix}, and more importantly it allows us to define the   {total cost} of an satisfying path (similar to Definition~4.5 in~\cite{smith2011optimal}):

The first summation in~\eqref{ch02:eq:plan-cost} represents the accumulated weights of transitions along the prefix and the second is the summation along the suffix. Note that $\gamma$ represents the relative weighting on the cost of transient response (the prefix) and steady response (the suffix) to the task specification.
where the  {prefix}  $\tau_{\text{pre}}$ is transversed only once and the  {suffix} $\tau_{\text{suf}}$ is repeated infinitely. A plan with this prefix-suffix structure has a finite representation as~\eqref{ch02:eq:prefix-suffix}. This structure is also called \emph{lasso-shapeed} in~\cite{schuppan2005shortest} with the stem $\tau_{\text{pre}}$ and the loop~$\tau_{\text{suf}}$.


\begin{definition}\label{ch02:def:feasible}
$\varphi$ is called \textbf{feasible} if there exists an infinite path $\tau$ of $\mathcal{T}_c$ that satisfies $\varphi$. \hfill $\blacktriangle$
\end{definition}

\subsubsection{B\"uchi automaton}\label{ch02:subsec:NBA}
Given an LTL formula $\varphi$ over  $AP$,  there exists a Nondeterministic B\"uchi automaton (NBA) over $2^{AP}$ corresponding to $\varphi$, denoted by $\mathcal{A}_{\varphi}$.
%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%
\begin{definition}\label{ch02:def:NBA}
The NBA $\mathcal{A}_{\varphi}$ is defined by a five-tuple:
\begin{equation}\label{ch02:eq:NBA}
\mathcal{A}_{\varphi}=(Q, \,2^{AP},\, \delta,\, Q_0,\, \mathcal{F}),
\end{equation}
where $Q$ is a finite set of states; $Q_0\subseteq Q$ is a set of initial states; $2^{AP}$ is the alphabet; $\delta: Q\times 2^{AP} \rightarrow 2^{Q}$ is a transition relation; $\mathcal{F}\subseteq Q$ is a set of accepting states.
\hfill $\blacktriangle$
\end{definition}
%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%

 An infinite run of the NBA is an infinite sequence of states that starts from an initial state and follows the transition relation. Namely, $r=q_0q_1q_2\cdots$, where $q_0\in Q_0$ and $q_{k+1}\in \delta(q_k,S)$ for some $S\in 2^{AP}$, $k=0,1,\cdots$. Moreover, $r$ is called \emph{accepting} if $\texttt{Inf}(r)\cap \mathcal{F}\neq \emptyset$, where $\texttt{Inf}(r)$ is the set of states that appear in $r$ infinitely often. We denote the successors of $q_m\in Q$ by $\texttt{Post}(q_m)=\{q_n\,|\, \exists S \in 2^{AP},\; q_n \in \delta(q_m,\, S)\}$.
%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%
\begin{definition}\label{ch02:def:cor-run}
Given an infinite word  $\sigma=S_0S_1S_2\cdots$ over $2^{AP}$, its \textbf{resulting run} in $\mathcal{A}_{\varphi}$ is denoted by $r_{\sigma}=q_0q_1q_2\cdots$, which satisfies: (i) $q_0 \in Q_0$; (ii) $q_{i+1} \in \delta(q_i,\, S_i)$, $\forall i=1,2,\cdots,\infty$.
Similar statement holds for a finite word $\bar{\sigma}=S_0S_1S_2\cdots S_{N+1}$.
% Similarly, given a finite word $\bar{\sigma}=S_0S_1S_2\cdots S_{N+1}$ over $2^{AP}$, its \textbf{resulting run} is denoted by $r_{\bar{\sigma}}=q_0q_1q_2\cdots q_N$, which satisfies: (i) $q_0 \in Q_0$; (ii) $q_{i+1} \in \delta(q_i,\, S_i)$, $\forall i=1,2,\cdots, N$.
\hfill $\blacktriangle$
\end{definition}
%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%
Note that since $\mathcal{A}_{\varphi}$ is nondeterministic, there may exist \emph{multiple} resulting runs of the same world. Denote by $\mathcal{L}_{\omega}(\mathcal{A}_{\varphi})$ the accepted language of $\mathcal{A}_{\varphi}$, which is the set of infinite words that result in an accepting run of $\mathcal{A}_{\varphi}$, i.e., $\mathcal{L}_w(\mathcal{A}_{\varphi})=\{\sigma\in (2^{AP})^{\omega}\,|\,r_{\sigma}\; \text{is an accepting run}\}$.
%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%
\begin{lemma}\label{ch02:lem:LTL-BA}
$\mathcal{L}_w(\mathcal{A}_{\varphi})=\textup{\texttt{Words}}(\varphi)$
\end{lemma}
\begin{proof}
See proof of Theorem 5.37 in~\cite{baier2008principles}
\end{proof}

The translation process from an LTL formula to its corresponding NBA can be done in time and space $2^{\mathcal{O}(|\varphi|)}$~\citep{baier2008principles}. However, there are fast translation algorithms~\citep{gastin2001fast}, which generates NBA with few states and transitions. Furthermore it is tedious to list all input alphabets for each transition, particularly given a large set of $AP$. Thus it is important to represent them in an compact and efficient manner.  The translation algorithm from~\cite{gastin2001fast} generates a boolean expression for each transition, which accepts all alphabets that enable this transition (as shown in Figure~\ref{ch02:fig:Buchi}). Binary decision diagrams (BDD) are well-known for their efficiency to represent and evaluate boolean functions~\citep{akers1978binary} . As a result, an NBA can be encoded symbolically and efficiently. More details can be found in Section~\ref{ch02:chapter:experiment}.

\subsection{Product B\"uchi automaton}\label{ch02:subsec:product}
The automaton-based model-checking  algorithm can be found in~\cite{vardi1986automata} and Algorithm 11 of~\cite{baier2008principles}. It is based on checking the emptiness of the product B\"uchi automaton. Since $\texttt{Words}(\varphi)=\mathcal{L}_w(\mathcal{A}_{\varphi})$ and $\texttt{trace}(\tau) \in \texttt{Trace}(\mathcal{T}_c)$, the original problem  is equivalent to finding the intersection  $\texttt{Trace}(\mathcal{T}_c)\cap \mathcal{L}_w(\mathcal{A}_{\varphi})$, which is actually the language of the product B\"uchi automaton $\mathcal{A}_p=\mathcal{T}_c \otimes \mathcal{A}_{\varphi}$, which accepts all runs that are valid  for $\mathcal{T}_c$ and at the same time satisfy $\varphi$.


It is important to mention that unlike the model-checking algorithm for verification, we do not negate the task specification before generating the associated NBA and the product automaton. This is because we are interested in the ``good'' behavior of the system that satisfies the specification, not the ``bad'' behavior  that satisfies the negated specification for the purpose of verification.
\begin{definition} \label{ch02:def:product}
The weighted product B\"uchi  automaton is defined by $\mathcal{A}_p=\mathcal{T}_c \otimes  \mathcal{A}_{\varphi}=(Q',\, \delta', \, Q_0',\, \mathcal{F}',\, W_p)
$, where
$Q'=\Pi \times Q =\{ \langle \pi,\, q \rangle\in Q' \, |\, \forall \pi\in \Pi$, \, $\forall q \in Q\}$;
$\delta': Q'\rightarrow 2^{Q'}$. $\langle \pi_j,\, q_n \rangle\in \delta'(\langle \pi_i,\, q_m \rangle)$ iff $(\pi_i,\,\pi_j) \in \longrightarrow_c$  and $q_n\in \delta(q_m,\,  L_c(\pi_i))$;
$Q_0'=\{ \langle \pi,\, q \rangle\, |\, \pi \in \Pi_0, \, \, q_0\in Q_0\}$, the set of initial states;
$\mathcal{F}'=\{ \langle \pi,\, q \rangle\, |\, \pi \in \Pi, \, q \in  \mathcal{F}\}$, the set of accepting states;
$W_p:Q'\times Q' \rightarrow \mathbb{R}^+$ is the weight function:
$
W_p(\langle \pi_i,\, q_m \rangle,\, \langle \pi_j,\, q_n \rangle)= W_c(\pi_i,\,\pi_j),
$
where $\langle \pi_j,\, q_n \rangle\in \delta'(\langle \pi_i,\, q_m \rangle)$. \hfill $\blacktriangle$
\end{definition}

Since $\mathcal{A}_p$ remains a B\"uchi automaton~\cite{baier2008principles}, its infinite run and its accepting condition can be defined similarly as $\mathcal{A}_{\varphi}$. An infinite run $R$ is called accepting if it intersects with the accepting set $\mathcal{F}'$ infinitely often. The successors of $q'_s$ are given by $\texttt{Post}(q'_s)=\{q'_g \,|\, q'_g \in \delta'(q'_s)\}$. Given a state $q'=\langle \pi, \, q\rangle \in Q'$, its projection on $\Pi$ is denoted by $q'|_{\Pi}=\pi$ and its projection on $Q$ is denoted by $q'|_{Q}=q$. Given an infinite run $R=q_0'q_1'q_2'\cdots$ of $\mathcal{A}_p$, its projection on $\Pi$ is denoted by $R|_{\Pi}=q_0'|_{\Pi}\,q_1'|_{\Pi}\, q_2'|_{\Pi}\cdots$ and its projection on $Q$ is denoted by $R|_{Q}=q_0'|_{Q}\,q_1'|_{Q}\, q_2'|_{Q}\cdots$.

%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%
\begin{lemma}\label{ch02:lem:non-empty}
If there exists an infinite path $\tau$ of $\mathcal{T}_c$ such that $\tau\models \varphi$, then \textbf{at least} an accepting run of $\mathcal{A}_p$ exists.
\end{lemma}
\begin{proof}
See the proof of Theorem 4.63 from~\cite{baier2008principles}.
\end{proof}

\begin{lemma}\label{ch02:lem:satisfy}
Given an accepting run $R$ of  $\mathcal{A}_p$, then $R|_{\Pi}\models \varphi$.
\end{lemma}
\begin{proof}
By definition, there exists an accepting state $q_f'\in \mathcal{F}'$  appearing in $R$ infinitely often. Thus $q_f'|_{Q}\in \mathcal{F}$  appears in $R|_{Q}$ infinitely often, yielding that $R|_{Q}$ is an accepting run. It can be easily shown that $R|_{Q}$ is one of the resulting runs of the trace of $R|_{\Pi}$. Thus $\texttt{trace}(R|_{\Pi}) \in \mathcal{L}_{\omega}(\varphi)=\texttt{Words}(\varphi)$, which implies $R|_{\Pi}\models \varphi$ by Definition~\ref{ch02:def:path-satisfy}.
\end{proof}

\subsubsection{Cost of an accepting run} \label{ch02:subsec:optimality}

We intended to find an infinite path $\tau$ of $\mathcal{T}_c$ with the prefix-suffix structure in~\eqref{ch02:eq:prefix-suffix}, such that ${\tau} \models \varphi$. The following lemma is important for the correctness.
%============
\begin{lemma}\label{ch02:lem:prefix-suffix}
If there exists an infinite path $\tau$ with the prefix-suffix structure and~$\tau \models \varphi$, then \textbf{at least} one accepting run of $\mathcal{A}_p$ exists with the prefix-suffix structure.
\end{lemma}
\begin{proof}
Since $\tau \models \varphi$, then $\sigma=\texttt{trace}(\tau) \in \texttt{Words}(\varphi)$. Furthermore as $\mathcal{L}_w(\mathcal{A}_{\varphi})=\texttt{Words}(\varphi)$ by~Lemma~\ref{ch02:lem:LTL-BA}, $\sigma \in \mathcal{L}_w(\mathcal{A}_{\varphi})$, meaning that the resulting run $r_\sigma$ in $\mathcal{A}_{\varphi}$ by~Definition~\ref{ch02:def:cor-run} is an accepting run of $\mathcal{A}_{\varphi}$.

Without loss of generality, let $\tau=\pi_0\pi_1\cdots \pi_i\cdots$ and $r_{\sigma}=q_0q_1\ldots q_j\cdots$. Now we prove that $R=\langle \pi_0,\,q_0\rangle \langle \pi_1,\,q_1\rangle\cdots \langle \pi_i,\,q_j\rangle\cdots$ is an accepting run of $\mathcal{A}_p$. First of all, $\langle \pi_0,\,q_1\rangle \in Q_0'$ as $\pi_0\in \Pi_0$ and $q_0 \in Q_0$. Secondly, $\langle \pi_{i+1},\,q_{j+1} \rangle \in \delta'(\langle \pi_i,\,q_j\rangle)$ as $(\pi_i,\,\pi_{i+1})\in \longrightarrow_c$ and $q_{i+1} \in \delta(q_{i},\, L_c(\pi_i))$. At last, since $r_{\sigma}$ is an accepting run of $\mathcal{A}_{\varphi}$, there exists an accepting state $q_f \in \mathcal{F}$ that appears in $r_{\sigma}$ infinitely often. Correspondingly, there exists at least one $\langle \pi,\, q_f\rangle $ that appears in $R$ infinitely often and  $\langle \pi,\, q_f\rangle \in \mathcal{F}'$, where $\pi$ may stand for one or several states in $\Pi$. Since $\mathcal{F}'$ is finite, there must be one accepting state $q'_f\in \mathcal{F}'$ that appears in $R$ infinitely often.

Then an accepting run with the prefix-suffix structure can be constructed by using the segment from $\langle\pi_0,\,q_0\rangle$ to $q'_f$ as prefix and the segment starting from $q'_f$ and back to $q'_f$ as the suffix, which completes the proof.
\end{proof}

Thus we could focus on the accepting runs of $\mathcal{A}_p$ with the  \emph{prefix-suffix} structure:
\begin{equation}\label{ch02:eq:prod-Apath}
\begin{split}
 R&= \langle R_{\textup{pre}},\, R_{\textup{suf}}\rangle=q'_0\, q'_1\cdots q'_f\,\left[\,q'_f \, q'_{f+1}\cdots \cdots q_n'\,\right]^{\omega}\\
&=\langle \pi_0, q_0\rangle\, \cdots \langle \pi_{f-1}, q_{f-1}\rangle \left[\, \langle \pi_f, q_f\rangle\langle \pi_{f+1}, q_{f+1}\rangle \cdots \cdots \langle \pi_n, q_n\rangle\, \right]^{\omega}\, ,
\end{split}
\end{equation}
where $q_0'=\langle \pi_0,\, q_0\rangle \in Q'_{0}$ and $q_f'=\langle \pi_f,\, q_f\rangle \in \mathcal{F}'$. Note that there are no correspondences among the subscripts. The  {prefix} part $R_{\textup{pre}}=(q'_0\, q'_1\cdots  q'_f)$ from an initial state $q'_0$ to one accepting state $q'_f$ that is executed only once while the  {suffix} part $R_{\textup{suf}}=(q'_f\, q'_{f+1}\cdots \cdots q'_n)$ from $q'_f$ back to itself that is repeated infinitely. Given the finite representation as~\eqref{ch02:eq:prod-Apath}, there is a  finite set of transitions appearing in $R$:
\begin{equation}\label{ch02:edgeR}
\textup{\texttt{Edge}}(R)=\{(q'_i,\, q'_{i+1}),\; i=0,1\cdots,(n-1)\}\cup\{(q'_n,\, q'_f)\}.
\end{equation}
The structure also allows us to define the {total cost} of $R$:
\begin{equation}\label{ch02:eq:prod-cost}
\begin{split}
\texttt{Cost}(R,\, \mathcal{A}_p)&=\sum_{i=0}^{f-1} W_{p}(q'_i,\, q'_{i+1})+\gamma \sum_{i=f}^{n-1} W_{p}(q'_i,\, q'_{i+1})\\
&=\sum_{i=0}^{f-1} W_{c}(\pi_i,\, \pi_{i+1})+\gamma \sum_{i=f}^{n-1} W_{c}(\pi_i,\, \pi_{i+1})\\
\end{split}
\end{equation}
where the first summation in~\eqref{ch02:eq:prod-cost} represents the accumulated weights of transitions along the prefix and the second  the suffix; $\gamma\geq 0$ is the relative weighting on the cost of transient response (the prefix) and steady response (the suffix).

%=======================
\begin{problem}\label{ch02:pro:opt-search}
Given the product automaton~$\mathcal{A}_p$, find its accepting run with the prefix-suffix structure that \textbf{minimizes} the cost defined by~\eqref{ch02:eq:prod-cost}. \hfill $\blacktriangle$
\end{problem}
We denote by $R_{\text{opt}}$ the solution for the above problem, as the optimal accepting run. Its corresponding plan is given by $\tau_{\textup{opt}}=R_{\text{opt}}|_{\Pi}$.

\subsection{Optimal run search}\label{ch02:subsec:alg-imp}
\label{ch02:sssec:plan_synthesis_algorithm}
In this part we present firstly two  methods to construct the product automaton and then the graph search algorithm to find the optimal accepting run.
\subsubsection{Full construction}\label{ch02:subsub:full-const}
%% full product
Since only static and fully-known workspace is considered, both $\mathcal{T}_c$ and $\varphi$ are time-invariant. Thus $\mathcal{A}_p$ can be constructed fully by Definition~\ref{ch02:def:product}.
In particular, given a FTS state $\pi_i \in \Pi$ and a NBA state $q_m \in Q$, the composed state $\langle \pi_i,\, q_m\rangle$ is added to $Q'$ if it is not in $Q'$ already. Then all transitions originated from $\langle \pi_i,\, q_m\rangle$ can be found by the definition of $\delta'$, namely $\forall \pi_j \in \texttt{Post}(\pi_i)$ and $\forall q_m \in \texttt{Post}(q_n)$, $\langle \pi_j,\, q_n\rangle \in \texttt{Post}(\langle \pi_i, \, q_m\rangle)$ if $q_n \in \delta(q_m,\, L(\pi_i))$.

%%%%%%%%%%%%%%%%%%%
\begin{figure}[t]
\centering
\includegraphics[height =2.1cm]{figures/ch02/alg.pdf}
\caption{For every pair of  $q_0'\in Q_0'$ (in blue) and $q_f'\in \mathcal{F}'$ (in red), the shortest path from $q_0'$ to $q_f'$ and the shortest cycle containing $q_f'$ are computed.}
\label{ch02:fig:accPath}
\end{figure}
%%%%%%%%%%%%%%%%%%%%


After constructing $\mathcal{A}_p$ fully, Algorithm~\ref{ch02:alg:optimal} takes as input arguments $\mathcal{A}_p$ and a set of starting states $S'\subseteq Q'$, which is set to $Q_0'$ by default, while it generates the optimal accepting run. Algorithm~\ref{ch02:alg:optimal} utilizes Dijkstra's algorithm~\citep{lavalle2006planning} for computing the shortest path from a single source node to a set of target nodes within a weighted  graph. In particular, function \texttt{DijksTargets}($\mathcal{A}_p$, $q_S'$, $Q_T'$)  computes shortest paths in $\mathcal{A}_p$ from source state $q_S'\in Q$ to every target state belonging to the set $Q_T'\in Q'$. Function \texttt{DijksCycle}($\mathcal{A}_p$, $q_S'$) is used to compute shortest cycle from the source state $q_S'$ back to itself. As shown in Figure~\ref{ch02:fig:accPath}, for each pair of initial and accepting states $(q_0',\, q_f')$ where  $q_0'\in Q_0'$ and $q_f' \in \mathcal{F}'$, the shortest path from $q_0'$ to $q_f'$ is obtained from line~1 of Algorithm~\ref{ch02:alg:optimal} where \texttt{DijksTargets}($\cdot$) is called while the shortest cycle containing $q'_f$ is obtained from line~2 of Algorithm~\ref{ch02:alg:optimal} where \texttt{DijksCycle}($\cdot$) is called.   At last, the pair $(q_{0,\star}',\, q_{f,\star}')$ that minimizes the total cost defined by~\eqref{ch02:eq:prod-cost} is chosen. Then the optimal accepting run is determined by setting its prefix as the shortest path from $q_{0,\star}'$ to $q_{f,\star}'$ and its suffix as shortest cycle containing $q_{f,\star}'$.
%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%
\begin{algorithm}[t]
\caption{Search for optimal run, \texttt{OptRun}().} \label{ch02:alg:optimal}
%\DontPrintSemicolon
\DontPrintSemicolon
\KwIn{$\mathcal{A}_p$, $S'=Q_0'$ by default}
\KwOut{$R_{\textup{opt}}$}
 1. If $Q_0'$ or $\mathcal{F}'$ is empty, construct $Q_0'$ or $\mathcal{F}'$ first.

 2. For each initial state $q_0'\in S'$, call \texttt{DijksTargets}($G$, $q_0'$, $\mathcal{F}'$).\;

 3. For each accepting state $q_f'\in \mathcal{F}'$, call \texttt{DijksCycle}($G$, $q_f'$).\;

 4. Find the pair of $(q_{0,\star}', q_{f,\star}')$ that minimizes the summed cost defined by~\eqref{ch02:eq:prod-cost}.\;

 5. Optimal accepting run $R_{\text{opt}}$, prefix: the shortest path from $q_{0,\star}'$ to $q_{f,\star}'$; suffix: the shortest cycle from $q_{f,\star}'$ and back to itself.\;

% \textbf{return} $R_{\textup{opt}}$
\end{algorithm}

The algorithm to compute~$\mathcal{A}_p$ has the complexity proportional to the sum of the number of states and transitions in $\mathcal{A}_p$. The worst-case  complexity of Algorithm~\ref{ch02:alg:optimal} is given by $\mathcal{O}(|\delta'| \cdot \log |Q'_0|\cdot (|Q'_0|+|\mathcal{F}'|))$) as essentially the Dijkstra algorithm~\cite{lavalle2006planning,korf1987planning} has to be called over $\mathcal{A}_p$ by the number of times equal to $|Q'_0|+|\mathcal{F}'|$.

\subsubsection{On-the-fly construction}\label{ch02:subsub:on-the-fly}
%% on the fly
Beside constructing $\mathcal{A}_p$ once for all, we would construct ${\mathcal{A}}_{p}$ on-the-fly along with the graph search Algorithm~\ref{ch02:alg:optimal} and even the revision Algorithm~\ref{ch03:alg:revise} introduced later. In other words, the states and transition relations of~${\mathcal{A}}_{p}$ are built ``on demand''. When the search algorithm visits any state~$q'_s\in Q'$, it constructs the adjacency relation of~$q'_s$ by iterating through all successors of~$q'_s$ and returns the corresponding transition~$q'_g \in \delta'(q'_s)$ along with its weight. Note that each state~$q'_s\in{Q}'$ is marked by the label ``visited'' or ``unvisited'', to indicate if the transitions originated from $q'_s$ have to be constructed. In particular, if $q_s'$ is marked ``visited'', it means that they have been constructed before and can be returned directly; if $q_s'$ is marked ``unvisited'' or $\pi_i$ belongs to $\widehat{\Pi}$, they need to be constructed by Definition~\ref{ch02:def:product} ; $\widehat{\Pi}$ will be defined in~\eqref{ch03:eq:widehat},
%which contains all regions whose labeling function or adjacency relation have changed. $\widehat{\Pi}$
which can be treated as $\emptyset$ for now.
%lines 5-12 construct the transitions by Definition~\ref{ch02:eq:product-weight} where $q'_s$'s successor $q_g'$ is added to $Q'$ as states to be visited~(line~10) by the planning algorithms introduced later.
Detailed implementation of the on-the-fly construction can be found in~\cite{guo2013cooperative}.



The markers ``visited'' and ``unvisited'' improve the computational efficiency as the adjacency relation of states that have been visited before can be returned directly, meaning that the results from previous planning iterations can be reused later; $\widehat{\Pi}$ provides a highly efficient way to maintain and update ${\mathcal{A}}_{p}$ in case of updates in $\mathcal{T}_c$, as later described in Algorithm~\ref{ch03:alg:revise} in Section~\ref{ch03:sec:task_relaxation_adaptation}. Then Algorithm~\ref{ch02:alg:optimal} can be easily modified such that it takes the adjacency relation of $\mathcal{A}_p$ as an input, instead of the fully-constructed~$\mathcal{A}_p$. Functions~\texttt{DijksTargets}($\cdot$) and~\texttt{DijksCycle}($\cdot$) still work in the same way since the Dijkstra algorithm only needs the adjacency relation in the breadth-first structure and the associated weight. Even though the worst-case computational complexity of Algorithm~\ref{ch02:alg:optimal} remains the same for these two different methods to construct~$\mathcal{A}_p$, the on-the-fly construction is essential for the partially-known and dynamic workspace that will be discussed in Section~\ref{ch03:sec:task_relaxation_adaptation}, where $\mathcal{T}_c$ has to be updated frequently.

\begin{example}\label{ch02:exam:complexity}
This example shows the general complexity of synthesizing a discrete motion and task plan using the proposed scheme. \subsection{Complexity under on-the-fly construction}
\label{ch02:sssec:complexity_on_the_fly}

The complexity analysis is divided into two cases. The first concerns the
 single-robot synthesis problem under on-the-fly product construction. Let
$n_{\varphi} \triangleq |\varphi|$, where $n_{\varphi}$ denotes the size of
 the LTL formula. Standard translation from LTL to B\"uchi automata yields
 \cite{baier2008principles,gastin2001fast}.
\begin{equation}
|Q| = 2^{\mathcal{O}(n_{\varphi})}, \qquad
|\delta| = 2^{\mathcal{O}(n_{\varphi})},
\label{ch02:eq:buchi-size-bound}
\end{equation}
where $Q$ and $\delta$ are the state and transition sets of
$\mathcal{B}_{\varphi}$ in Definition~\ref{ch02:def:NBA}. Once
$\mathcal{T}$ and $\mathcal{B}_{\varphi}$ are fixed, the full product
 automaton in Definition~\ref{ch02:def:product} satisfies
$|X| = |S|\,|Q|$ and
$|\rightarrow_{\mathcal{P}}| =
\mathcal{O}\bigl(|\rightarrow|\,|\delta|\bigr)$, where $|S|$ and
$|\rightarrow|$ denote the number of states and transitions in $\mathcal{T}$.

Under on-the-fly construction, only the subset of product states and transitions
 requested by the graph search is generated. Let
$X_{\mathrm{vis}} \triangleq$ the set of visited product states during the
 search, where $X_{\mathrm{vis}} \subseteq X$, and let
$\rightarrow_{\mathrm{vis}} \triangleq
\{(x,x') \in \rightarrow_{\mathcal{P}} \mid x \in X_{\mathrm{vis}}\}$,
 where $\rightarrow_{\mathrm{vis}}$ is the induced set of generated product
 transitions. If shortest-path and shortest-cycle computations are performed
 from all candidate initial and accepting states, then the effective synthesis
 time is bounded by
\begin{equation}
T_{\mathrm{otf}} =
\mathcal{O}\Bigl(
\bigl(|X_{0}| + |F_{\mathrm{vis}}|\bigr)
|\rightarrow_{\mathrm{vis}}|
\log |X_{\mathrm{vis}}|
\Bigr),
\label{ch02:eq:otf-time}
\end{equation}
where $F_{\mathrm{vis}} \triangleq F_{\mathcal{P}} \cap X_{\mathrm{vis}}$
 denotes the accepting product states that are actually generated. The
 corresponding memory requirement is
$M_{\mathrm{otf}} =
\mathcal{O}\bigl(|X_{\mathrm{vis}}| + |\rightarrow_{\mathrm{vis}}|\bigr)$.

The worst-case asymptotic bound of the on-the-fly procedure coincides with that
 of full construction, since $X_{\mathrm{vis}} = X$ and
$\rightarrow_{\mathrm{vis}} = \rightarrow_{\mathcal{P}}$ when the search
 explores the entire product graph. However, in practice the on-the-fly
 strategy can be substantially cheaper, because many product states need not be
 generated if they are unreachable or irrelevant to the optimal accepting run.
 This distinction mirrors the discussion in the Chapter 3 draft on full versus
 on-the-fly construction.

\subsection{Multi-robot scalability}
\label{ch02:sssec:multi_robot_scalability}

The above complexity is still acceptable for a single robot when both the
 workspace abstraction and the specification size remain moderate. The situation
 changes significantly if the same method is applied directly to a system with
 multiple robots through a centralized team model. Consider $N$ robots with
 individual transition systems
$\mathcal{T}_{i} \triangleq
\bigl(S_{i},S_{i,0},\rightarrow_{i},\mathsf{AP}_{i},L_{i},w_{i}\bigr)$,
 where $i \in \{1,\ldots,N\}$. A direct joint abstraction leads to the team
 state space $S_{\mathrm{team}} \triangleq \prod_{i=1}^{N} S_{i}$, where each
 joint state records the simultaneous configuration of all robots. Consequently,
$|S_{\mathrm{team}}| = \prod_{i=1}^{N} |S_{i}|$, where the growth is
 multiplicative in the number of robots.

If a team-level temporal specification $\Phi$ is translated into a B\"uchi
 automaton $\mathcal{B}_{\Phi} \triangleq
\bigl(Q_{\Phi},Q_{\Phi,0},2^{\mathsf{AP}_{\mathrm{team}}},
\delta_{\Phi},F_{\Phi}\bigr)$, then the corresponding centralized product
 space becomes
\begin{equation}
|X_{\mathrm{team}}| =
|Q_{\Phi}| \prod_{i=1}^{N} |S_{i}|,
\label{ch02:eq:team-product-cardinality}
\end{equation}
where the blow-up arises from both the coupled motion state and the task
 automaton. The transition set also grows combinatorially, since every
 synchronized or asynchronously compatible joint move contributes a team-level
 transition. Therefore, both the graph-search cost and the memory demand inherit
 the multiplicative growth in \eqref{ch02:eq:team-product-cardinality}.

The implications are immediate. Even when the local models are modest, the
 direct centralized construction becomes computationally prohibitive as the
 number of robots increases. This observation explains why single-robot methods
 cannot be transferred naively to the multi-robot setting. Instead, the later
 chapters will exploit local specifications, compositional abstractions, and
 coordination mechanisms in order to avoid the full centralized blow-up in
 \eqref{ch02:eq:team-product-cardinality}. Related scalability issues have also been
 noted in the literature on multi-robot temporal-logic planning and deployment;
 see, for example, \cite{kloetzer2010automatic,chen2012formal,
 kloetzer2011multi,guo2015multi}.

\section{Case Study}
\label{ch02:sec:case_study}
\label{ch02:chapter:experiment}

This section illustrates the proposed single-robot framework on a TIAGo-based
 simulation example. The implementation combines three components: a finite
 transition model of the robot motion, an LTL planner that synthesizes a
 satisfying symbolic plan, and a FlexBE execution layer that dispatches the
 corresponding navigation and manipulation behaviors. The example is adapted
 from the publicly available \texttt{tiago\_ltl\_flexbe} implementation,
 which builds on the P\_MAS\_TG planner and related LTL-based robotic
 planning tools \cite{package,LTLfirst,guo2013motion}.

\subsection{Workspace and abstract model}
\label{ch02:sssec:case_study_workspace_model}

The workspace consists of three regions of interest associated with navigation
 waypoints. Let $P_{\mathrm{cs}} \triangleq \{p_{1},p_{2},p_{3}\}$, where
$p_{1} = (0.57,-10.01,0.95)$,
$p_{2} = (-3.25,-0.31,0.82)$, and
$p_{3} = (-2.89,-9.01,0.52)$. The triplets denote planar position and heading.
 The associated region alphabet is
$\mathsf{AP}_{\mathrm{reg}}^{\mathrm{cs}} \triangleq
\{r_{1},r_{2},r_{3}\}$, where $r_i$ indicates that the robot occupies the
 region centered at $p_i$.

Besides navigation, the implementation supports discrete action labels through
 the TIAGo \texttt{play\_motion} interface. For the present example, only two
 actions are needed, namely
$\mathsf{AP}_{\mathrm{act}}^{\mathrm{cs}} \triangleq
\{\mathsf{pick},\mathsf{reach}\}$, where
$\mathsf{pick} \triangleq \texttt{pick\_from\_floor}$ and
$\mathsf{reach} \triangleq \texttt{reach\_max}$. The complete proposition
 alphabet is therefore
$\mathsf{AP}^{\mathrm{cs}} \triangleq
\mathsf{AP}_{\mathrm{reg}}^{\mathrm{cs}} \cup
\mathsf{AP}_{\mathrm{act}}^{\mathrm{cs}}$, which instantiates the general
 alphabet introduced in Section~\ref{ch02:sec:system_abstraction_task_specification}.

The motion abstraction is fully connected in the implementation. Hence, every
 region is reachable from every other region. The motion cost is proportional to
 the Euclidean waypoint distance and is given by
$w_{\mathrm{mot}}^{\mathrm{cs}}(s_i,s_j) \triangleq
0.1\bigl(\|(p_i)_x - (p_j)_x,\,(p_i)_y - (p_j)_y\|_2 + 10^{-3}\bigr)$,
 where $(p_i)_x$ and $(p_i)_y$ denote the planar coordinates of waypoint $p_i$.
 The action abstraction is represented by
$w_{\mathrm{act}}^{\mathrm{cs}}(\alpha) \triangleq 10$ for each
$\alpha \in \{\mathsf{pick},\mathsf{reach}\}$.

To account for both navigation and discrete actions, the single-robot planning
 model is taken as the motion-action transition system
\begin{equation}
\mathcal{T}_{\mathrm{cs}} \triangleq
\bigl(S_{\mathrm{cs}},S_{\mathrm{cs},0},
\rightarrow_{\mathrm{cs}},\mathsf{AP}^{\mathrm{cs}},
L_{\mathrm{cs}},w_{\mathrm{cs}}\bigr),
\label{ch02:eq:case-study-full-ts}
\end{equation}
where
$S_{\mathrm{cs}} \triangleq
S_{\mathrm{mot}}^{\mathrm{cs}} \times
\{\mathsf{none},\mathsf{pick},\mathsf{reach}\}$ and
$S_{\mathrm{cs},0} \triangleq \{(s_{1},\mathsf{none})\}$. Here,
$\mathsf{none}$ denotes the absence of a discrete action, and the robot starts
 from the waypoint associated with $r_1$. The labeling map satisfies
$L_{\mathrm{cs}}(s_i,\mathsf{none}) \triangleq \{r_i\}$ and
$L_{\mathrm{cs}}(s_i,\alpha) \triangleq \{r_i,\alpha\}$ for each
$\alpha \in \{\mathsf{pick},\mathsf{reach}\}$. Navigation transitions are
 of the form $(s_i,\beta) \rightarrow_{\mathrm{cs}} (s_j,\mathsf{none})$,
 where $s_i$ and $s_j$ are motion-adjacent states and
$\beta \in \{\mathsf{none},\mathsf{pick},\mathsf{reach}\}$, while action
 transitions are of the form
$(s_i,\mathsf{none}) \rightarrow_{\mathrm{cs}} (s_i,\alpha)$. Their weights
 are inherited from $w_{\mathrm{mot}}^{\mathrm{cs}}$ and
$w_{\mathrm{act}}^{\mathrm{cs}}$, respectively.

\subsection{Task specification and synthesized plan}
\label{ch02:sssec:case_study_task_plan}

The case-study task requires the robot to go to region $r_2$, execute the
 pickup action there, and then reach region $r_3$ and execute the reaching
 action. The corresponding LTL formula is
\begin{equation}
\varphi_{\mathrm{cs}} \triangleq
\Diamond \Bigl(
(r_{2} \wedge \mathsf{pick})
\wedge
\Diamond (r_{3} \wedge \mathsf{reach})
\Bigr),
\label{ch02:eq:case-study-task}
\end{equation}
where \eqref{ch02:eq:case-study-task} is the symbolic counterpart of the task
 expression used in the implementation. Applying the synthesis procedure from
 Section~\ref{ch02:sec:automata_synthesis_hybrid_control} to
$\mathcal{T}_{\mathrm{cs}}$ and \eqref{ch02:eq:case-study-task} yields a product
 automaton of the form Definition~\ref{ch02:def:product}, and the corresponding
 optimal accepting run is obtained in prefix--suffix form as in
 Subsection~\ref{ch02:sssec:plan_synthesis_algorithm}.

For the present instance, the optimal discrete prefix is
\begin{equation}
\rho_{\mathrm{cs}}^{\star,\mathrm{pre}} \triangleq
(s_{1},\mathsf{none})
(s_{2},\mathsf{none})
(s_{2},\mathsf{pick})
(s_{3},\mathsf{none})
(s_{3},\mathsf{reach})
(s_{3},\mathsf{none}),
\label{ch02:eq:case-study-prefix}
\end{equation}
where the first motion step drives the robot from $r_1$ to $r_2$, the second
 step executes the pickup action at $r_2$, the third step drives the robot from
$r_2$ to $r_3$, and the fourth step executes the reaching action at $r_3$. A
 trivial accepting suffix is given by
$\rho_{\mathrm{cs}}^{\star,\mathrm{suf}} \triangleq
(s_{3},\mathsf{none})^{\omega}$, where the robot remains in the terminal
 region after task completion. Since the motion graph is fully connected, the
 only nontrivial cost comparison concerns the ordering of the required visits
 and actions.

Using the motion and action costs defined in
Subsubsection~\ref{ch02:sssec:case_study_workspace_model}, the prefix cost induced
 by \eqref{ch02:eq:case-study-prefix} is
\begin{equation}
J_{\mathrm{cs}}^{\mathrm{pre}}
\triangleq
w_{\mathrm{mot}}^{\mathrm{cs}}(s_{1},s_{2})
+
w_{\mathrm{act}}^{\mathrm{cs}}(\mathsf{pick})
+
w_{\mathrm{mot}}^{\mathrm{cs}}(s_{2},s_{3})
+
w_{\mathrm{act}}^{\mathrm{cs}}(\mathsf{reach})
=
21.9134,
\label{ch02:eq:case-study-prefix-cost}
\end{equation}
where the numerical value follows from the waypoint coordinates. The plan in
 \eqref{ch02:eq:case-study-prefix} is therefore optimal for the instantiated model,
 since any feasible alternative would either preserve the same action cost while
 adding extra motion, or violate the ordering imposed by
 \eqref{ch02:eq:case-study-task}.

\subsection{Hybrid execution and discussion}
\label{ch02:sssec:case_study_execution_discussion}

The symbolic plan in \eqref{ch02:eq:case-study-prefix} is executed through the
 hybrid control structure described in Section~
ef{ch02:sec:hybrid}. In implementation
 terms, the continuous execution consists of the sequence
\begin{equation}
\texttt{goto}(r_{2})
\rightarrow
\texttt{pick\_from\_floor}
\rightarrow
\texttt{goto}(r_{3})
\rightarrow
\texttt{reach\_max},
\label{ch02:eq:case-study-execution-sequence}
\end{equation}
where \eqref{ch02:eq:case-study-execution-sequence} is the implementation-level
 counterpart of the discrete plan in \eqref{ch02:eq:case-study-prefix}. Under the
 assumptions of the hybrid execution framework, the resulting
 execution satisfies \eqref{ch02:eq:case-study-task}.

This case study illustrates how the abstract constructions introduced in
 Sections~\ref{ch02:sec:system_abstraction_task_specification} and
 \ref{ch02:sec:automata_synthesis_hybrid_control} can be instantiated in a practical
 single-robot setting. The example also shows that, once the workspace and the
 available actions have been encoded as a finite transition system, complex
 navigation-and-action tasks can be expressed compactly in temporal logic and
 executed systematically through a hybrid planning-and-control architecture.

\label{ch02:sec:ch02_summary}
