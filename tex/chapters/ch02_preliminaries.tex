\chapter{Preliminaries for Single-Robot Temporal Logic Task Planning}
\label{chap:preliminaries}

This chapter presents the nominal single-robot framework for integrated motion
and task planning. Robot motion is abstracted as a finite-state transition
system, temporal specifications are interpreted over the resulting symbolic
states, and automata-based synthesis computes a cost-aware plan. The plan is
then refined into a hybrid controller that realizes the prescribed task in the
continuous workspace.

%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%
%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%

\section{Abstraction of Robot Motion}
\label{ch02:sec:system_abstraction_task_specification}

This section constructs the discrete motion abstraction used for formal task
specification and plan synthesis. The workspace is first partitioned into
finitely many regions; the robot dynamics and available navigation laws then
determine which inter-region motions are feasible; and the resulting motion
capability is encoded as a weighted finite-state transition system. This model
serves as the nominal substrate for the temporal-logic framework used below.

\subsection{Workspace model}
\label{ch02:subsec:workspacemodel}

Consider a bounded workspace
$\mathcal{W}_0 \subset \mathbb{R}^{n}$.
Let
$\Pi \triangleq \{\pi_1,\cdots,\pi_N\}$
be a finite collection of regions of interest, where each
$\pi_i \subset \mathcal{W}_0$.
The set $\Pi$ is assumed to partition the workspace, so the regions jointly
cover $\mathcal{W}_0$ and are pairwise disjoint. Each robot position is
therefore assigned to one symbolic state. Atomic propositions describe the
robot and workspace properties relevant to the task. The first class records
robot location. Define
$AP_r \triangleq \{a_{r,1},\cdots,a_{r,N}\}$,
where each proposition $a_{r,i}$ indicates whether the robot is located in
region $\pi_i$. Its truth value is given by:
\begin{equation}
a_{r,i} \triangleq
\begin{cases}
\top, & \text{if the robot is located in } \pi_i, \\
\bot, & \text{otherwise},
\end{cases}
\label{ch02:eq:region_proposition}
\end{equation}
where the truth value in \eqref{ch02:eq:region_proposition} can be evaluated
from the output of a localization or positioning module.
Because $\Pi$ is a partition of $\mathcal{W}_0$, exactly one proposition from
$AP_r$ is true at every instant. This property is essential for obtaining an
unambiguous symbolic description of the robot motion. In addition to the
location propositions, task-dependent properties of the workspace are encoded by
a second set
$AP_p \triangleq \{a_{p,1},\cdots,a_{p,M}\}$.
These propositions may express, for example, the presence of an object, the
availability of a resource, or the designation of a service location. The full
set of atomic propositions is then defined by
$AP \triangleq AP_r \cup AP_p$.

\begin{definition}
\label{ch02:def:labeling}
A labeling function is a map
$L : \Pi \to 2^{AP}$
that assigns to each region $\pi_i \in \Pi$ the set of atomic propositions that
hold in that region. In particular,
$a_{r,i} \in L(\pi_i)$
for every $i \in \{1,\cdots,N\}$.
\end{definition}

Definition~\ref{ch02:def:labeling} attaches semantic information to each
symbolic state. A robot motion through the partition then induces a sequence of
truth assignments, which is later interpreted against temporal logic formulas.
The labeling is assumed uniform within each region; if a property holds only on
a subset of a region, the region should be refined until the truth assignment is
cell-wise uniform. Regions should also admit compact geometric descriptions,
such as rectangles, disks, or polygons. Standard procedures, including Delaunay
triangulation~\citep{lee1980two} and Voronoi-based
decomposition~\citep{lavalle2006planning}, may generate such partitions, but
this purely geometric step does not yet encode dynamics or controllers.


\subsection{Robot dynamics and navigation controller}
\label{ch02:subsec:robotdynamics}

The robot is modeled by the continuous-time control system below:
\begin{equation}
\dot{x}(t) = f\bigl(x(t),u(t)\bigr),
\label{ch02:eq:dynamic}
\end{equation}
where $x(t) \in \mathbb{R}^{n}$ is the system state,
$u(t) \in \mathbb{R}^{m}$ is the control input, and
$f : \mathbb{R}^{n} \times \mathbb{R}^{m} \to \mathbb{R}^{n}$
is assumed to be Lipschitz continuous~\citep{khalil2002nonlinear}.
The Lipschitz condition ensures deterministic trajectories for admissible
initial states and controls. The construction is stated for a general nonlinear
model, although the examples mainly use single-integrator and unicycle
dynamics. Given the partition in Section~\ref{ch02:subsec:workspacemodel}, the
relevant question is not mere geometric adjacency, but whether the robot can be
steered between regions under \eqref{ch02:eq:dynamic}. The transition relation
is therefore defined in a control-driven manner.

\begin{definition}
\label{ch02:def:transition}
Let $\pi_i,\pi_j \in \Pi$. A control-driven transition from $\pi_i$ to $\pi_j$
exists if there is an admissible navigation controller
$\mathcal{U} : \mathbb{R}^{n} \times \Pi \times \Pi \to \mathbb{R}^{m}$ such
that the control input is given by:
\begin{equation}
u(t) \triangleq \mathcal{U}\bigl(x(t),\pi_i,\pi_j\bigr),
\quad t \in [t_s,t_f],
\label{ch02:eq:control}
\end{equation}
where the corresponding closed-loop trajectory of
\eqref{ch02:eq:dynamic} satisfies
$x(t_s) \in \pi_i$,
$x(t_f) \in \pi_j$,
and
$x(t) \in \pi_i \cup \pi_j$
for all $t \in [t_s,t_f]$.
\end{definition}

Definition~\ref{ch02:def:transition} requires both controller realizability and
confinement to the source--target region pair. The latter prevents a symbolic
transition from implicitly using intermediate regions absent from the discrete
plan. Thus the abstraction is not purely geometric: a symbolic transition is
included only when a concrete controller can implement it in the continuous
domain.
The existence of a suitable navigation law depends on both the robot model and
the workspace decomposition. Representative constructions include
potential-field navigation for differential-drive robots through prescribed
facets~\citep{lindemann2006real} and navigation functions for provably correct
point-to-point motion in sphere worlds~\citep{koditschek1990robot}, with
extensions to single- and multi-robot settings
\citep{loizou2008density,loizou2002closed,dimarogonas2007decentralized,
dimarogonas2005decentralized}. Since a controller is rarely compatible with an
arbitrary partition, the geometry may require refinement or reshaping, yielding
an over- or under-approximation~\citep{saidi1999abstract} before the
abstraction is complete.


\subsection{Control-driven weighted transition system}
\label{ch02:subsec:FTS}

The preceding ingredients define the symbolic motion model. Each region
$\pi_i \in \Pi$ becomes a discrete state, and admissible state changes are
exactly those induced by Definition~\ref{ch02:def:transition}.

\begin{definition}
\label{ch02:def:FTS}
The control-driven weighted finite-state transition system is the tuple below:
\begin{equation}
\mathcal{T}_c \triangleq
\bigl(\Pi,\longrightarrow_c,\Pi_0,AP,L_c,w_c\bigr),
\label{ch02:eq:fts}
\end{equation}
where $\Pi$ is the finite state set induced by the workspace partition,
$\longrightarrow_c \subseteq \Pi \times \Pi$ is the control-driven transition
relation,
$\Pi_0 \subseteq \Pi$ is the set of initial states,
$AP$ is the set of atomic propositions,
$L_c : \Pi \to 2^{AP}$ is the labeling function inherited from
Definition~\ref{ch02:def:labeling}, and
$w_c : \longrightarrow_c \to \mathbb{R}_{\geq 0}$ assigns a nonnegative cost to
each enabled transition.
\end{definition}

The weighted transition system in \eqref{ch02:eq:fts} is the discrete model on
which tasks are interpreted. The weight $w_c$ may represent travel time, path
length, control effort, energy, or another metric, enabling synthesis to account
for both satisfaction and cost. The model is assumed to have no terminal states,
as is natural for the infinite-horizon specifications considered later. For each
state
$\pi_i \in \Pi$, define the successor set
$\texttt{Post}(\pi_i) \triangleq
\{\pi_j \in \Pi \mid \pi_i \longrightarrow_c \pi_j\}$.
An infinite path of $\mathcal{T}_c$ is then an infinite sequence
$\tau = \pi_0\pi_1\pi_2\cdots$
such that
$\pi_0 \in \Pi_0$
and
$\pi_{k+1} \in \texttt{Post}(\pi_k)$
for all $k \geq 0$.
Every path induces a label sequence. The trace generated by
$\tau$ is
$\texttt{trace}(\tau) \triangleq
L_c(\pi_0)L_c(\pi_1)L_c(\pi_2)\cdots$, and the trace language is
$\texttt{Trace}(\mathcal{T}_c) \triangleq
\{\texttt{trace}(\tau) \mid
\tau \text{ is an infinite path of } \mathcal{T}_c\}$.
Temporal logic formulas are evaluated over these infinite words rather than
directly over continuous trajectories. Ultimately periodic paths may be written
compactly using the $\omega$-operator~\citep{baier2008principles}. The
transition system $\mathcal{T}_c$ is fully known if it faithfully captures the
workspace and motion capabilities, and static if it does not vary with time.
These constructions fix the nominal symbolic model for the chapter. Workspace
partitioning, control-driven reachability, and weighted transitions yield a
finite abstraction grounded in dynamics rather than geometry alone.

\section{Task Specification as LTL Formulas}
\label{ch02:sec:LTL}

A formal task language must express logical conditions, temporal ordering, and
recurrent requirements without ambiguity. This chapter uses linear-time
temporal logic, or LTL, because it captures common motion and surveillance
tasks and its infinite-word semantics matches the executions of
$\mathcal{T}_c$ introduced in Section~\ref{ch02:subsec:FTS}.

\subsection{Syntax and semantics}
\label{ch02:subsec:syntax}

The syntax of LTL is defined over the atomic proposition set $AP$. The formulas
are constructed recursively from atomic propositions and temporal operators.

\begin{definition}
\label{ch02:def:ltl_syntax}
An LTL formula over $AP$ is generated by the grammar
\begin{equation}
\varphi ::= \top \mid a \mid \neg \varphi \mid \varphi_1 \vee \varphi_2
\mid \bigcirc \varphi \mid \varphi_1 \,\mathsf{U}\, \varphi_2,
\label{ch02:eq:LTLsyntax}
\end{equation}
where $a \in AP$, $\neg$ for negation,
$\vee$ for disjunction, $\bigcirc$ for the next operator, and
$\mathsf{U}$ for the until operator.
\end{definition}

Other task operators, including conjunction, implication, always, and
eventually, are derived from these primitives in the standard way; see
Chapter~5 of \citep{baier2008principles}. LTL semantics is defined over
infinite words on $2^{AP}$, whose symbols record the propositions true at each
discrete instant.

\begin{definition}
\label{ch02:def:word_semantics}
An infinite word over $2^{AP}$ is a sequence
$\sigma \triangleq S_0 S_1 S_2 \cdots$, where
$S_k \in 2^{AP}$ for every $k \geq 0$. The satisfaction relation
$(\sigma,k) \models \varphi$ is defined recursively by
\begin{equation}
\begin{aligned}
(\sigma,k) \models a
&\iff a \in S_k, \\
(\sigma,k) \models \neg \varphi
&\iff (\sigma,k) \not\models \varphi, \\
(\sigma,k) \models \bigcirc \varphi
&\iff (\sigma,k+1) \models \varphi, \\
(\sigma,k) \models \varphi_1 \vee \varphi_2
&\iff (\sigma,k) \models \varphi_1
\text{ or } (\sigma,k) \models \varphi_2, \\
(\sigma,k) \models \varphi_1 \,\mathsf{U}\, \varphi_2
&\iff \exists j \geq k \text{ such that }
(\sigma,j) \models \varphi_2 \\
&\quad \text{and }
(\sigma,\ell) \models \varphi_1
\text{ for every } \ell \in \{k,\cdots,j-1\},
\end{aligned}
\label{ch02:eq:ltl_semantics}
\end{equation}
where $a \in AP$, $k \in \mathbb{N}$, and $S_k$ is the set of propositions
satisfied at position $k$ of the word.
\end{definition}

A word $\sigma$ satisfies $\varphi$ if $(\sigma,0)\models \varphi$, written
$\sigma \models \varphi$. The language generated by $\varphi$ is
$\texttt{Words}(\varphi) \triangleq
\{\sigma \in (2^{AP})^{\omega} \mid \sigma \models \varphi\}$.
Since every infinite path $\tau$ of $\mathcal{T}_c$ generates the word
$\texttt{trace}(\tau)$, the semantics in \eqref{ch02:eq:ltl_semantics} applies
directly to traces of the control-driven weighted transition system.

\begin{definition}
\label{ch02:def:path-satisfy}
Let $\tau$ be an infinite path of $\mathcal{T}_c$, and let $\varphi$ be an LTL
formula over $AP$. The path $\tau$ is said to satisfy $\varphi$ if
$\texttt{trace}(\tau) \models \varphi$. In that case, one writes
$\tau \models \varphi$. A satisfying path is also called a plan for
$\varphi$.
\end{definition}

LTL expresses common robotic tasks in a unified form: safety as
$\square \neg \psi$, ordered visitation as
$\Diamond(\psi_1 \wedge \Diamond(\psi_2 \wedge \Diamond \psi_3))$, response as
$\square(\psi_1 \Rightarrow \Diamond \psi_2)$, and persistent surveillance as
$\square \Diamond \psi$. A useful subclass is syntactically co-safe LTL, or
sc-LTL~\citep{kupferman2001model}, whose satisfaction can be certified from a
finite prefix. This property is useful for goal-reaching and sequencing tasks;
the standard syntactic characterization follows
\citep{kupferman2001model,fainekos2009robustness}.

\subsection{Problem formulation}
\label{ch02:sec:single-problem}

The preceding semantics determines whether an infinite path satisfies a task
formula. Synthesis additionally requires computing a concrete path that can be
executed through a continuous control law of the form \eqref{ch02:eq:control}.
The path is therefore represented in prefix-suffix, or lasso,
form~\citep{schuppan2005shortest}, with a transient part executed once and a
recurrent part repeated indefinitely:
\begin{equation}
\tau \triangleq \tau_{\textup{pre}}
[\tau_{\textup{suf}}]^{\omega},
\label{ch02:eq:prefix-suffix}
\end{equation}
where $\tau_{\textup{pre}}$ is a finite prefix executed once, and
$\tau_{\textup{suf}}$ is a finite suffix repeated infinitely.
The representation in \eqref{ch02:eq:prefix-suffix} gives a finite description
of an infinite execution, matches the accepting structure of the product
automaton introduced below, and separates transient behavior from recurrent
behavior for cost and execution analysis.

\begin{definition}
\label{ch02:def:feasible}
An LTL formula $\varphi$ is said to be feasible on $\mathcal{T}_c$ if there
exists an infinite path $\tau$ of $\mathcal{T}_c$ such that
$\tau \models \varphi$.
\end{definition}

Definition~\ref{ch02:def:feasible} characterizes symbolic solvability on the
transition system. If $\varphi$ is feasible, a satisfying path must be
synthesized and then implemented in the continuous workspace through hybrid
control.
The task language, trace semantics, and nominal planning problem induced by
$(\mathcal{T}_c,\varphi)$ are now fixed. The remaining question is constructive:
how to synthesize a satisfying, cost-aware plan and realize it by hybrid
execution.

\begin{problem}
\label{ch02:prob:nominal}
Given the control-driven weighted transition system $\mathcal{T}_c$ and an LTL
formula $\varphi$ over $AP$, determine a plan of the form
\eqref{ch02:eq:prefix-suffix} such that $\tau \models \varphi$, and construct a
hybrid control strategy based on \eqref{ch02:eq:control} that executes the
resulting symbolic plan in the continuous workspace.
\end{problem}

\section{Hybrid Controller Synthesis}
\label{ch02:sec:solution}

This section solves Problem~\ref{ch02:prob:nominal} through the standard
automata-theoretic synthesis pipeline specialized to $\mathcal{T}_c$. The task
formula is translated into a Nondeterministic B\"uchi automaton, composed with
$\mathcal{T}_c$ as a weighted product automaton, searched for an accepting
prefix-suffix run, and finally executed by a hybrid controller.

\subsection{B\"uchi automaton associated with the task formula}
\label{ch02:subsec:NBA}

Given an LTL formula $\varphi$ over $AP$, a Nondeterministic B\"uchi automaton
accepts exactly the infinite words satisfying $\varphi$, thereby linking
temporal logic to graph-based synthesis.

\begin{definition}
\label{ch02:def:NBA}
The Nondeterministic B\"uchi automaton associated with $\varphi$ is denoted by
$\mathcal{A}_{\varphi}$ and defined by:
\begin{equation}
\mathcal{A}_{\varphi} \triangleq
\left(Q,\,2^{AP},\,\delta,\,Q_0,\,\mathcal{F}\right),
\label{ch02:eq:NBA}
\end{equation}
where $Q$ is a finite set of automaton states, $2^{AP}$ is the input alphabet,
$\delta : Q \times 2^{AP} \to 2^{Q}$ is the transition map,
$Q_0 \subseteq Q$ is the set of initial states, and
$\mathcal{F} \subseteq Q$ is the set of accepting states.
\end{definition}

The automaton in \eqref{ch02:eq:NBA} evolves over infinite words on
$2^{AP}$. Acceptance is defined by infinite repetition of designated states,
the standard B\"uchi condition.
Since $\mathcal{A}_{\varphi}$ is nondeterministic, a word may induce several
runs; acceptance requires at least one accepting run.

\begin{definition}
\label{ch02:def:buchi_run}
Let
$\sigma \triangleq S_0S_1S_2\cdots \in (2^{AP})^{\omega}$.
A resulting run of $\sigma$ in $\mathcal{A}_{\varphi}$ is an infinite sequence
$r_{\sigma} \triangleq q_0q_1q_2\cdots$ such that
$q_0 \in Q_0$
and
$q_{k+1} \in \delta(q_k,S_k)$
for every $k \geq 0$. The run is accepting if
$\texttt{Inf}(r_{\sigma}) \cap \mathcal{F} \neq \emptyset$,
where $\texttt{Inf}(r_{\sigma})$ denotes the set of states that appear
infinitely often in $r_{\sigma}$.
\end{definition}

\begin{definition}
\label{ch02:def:buchi_language}
The accepted language of $\mathcal{A}_{\varphi}$ is
$\mathcal{L}_{\omega}(\mathcal{A}_{\varphi})
\triangleq
\{
\sigma \in (2^{AP})^{\omega}
\,|\,
\text{some resulting run}$
$\text{ of $\sigma$ in $\mathcal{A}_{\varphi}$ is accepting}\}$.
\end{definition}


\begin{lemma}
\label{ch02:lem:LTL-BA}
The accepted language of $\mathcal{A}_{\varphi}$ coincides with the language of
$\varphi$, namely
$\mathcal{L}_{\omega}(\mathcal{A}_{\varphi})
= \textup{\texttt{Words}}(\varphi)$.
\end{lemma}

\begin{proof}
The statement follows from the standard translation theorem from LTL to
B\"uchi automata. A proof is given in Theorem~5.37 of
\citep{baier2008principles}.
\end{proof}

LTL-to-B\"uchi translation is worst-case exponential in formula size
\citep{baier2008principles}, although optimized procedures often yield much
smaller automata. A common construction is given in~\citep{gastin2001fast}.
Implementations usually label transitions by Boolean formulas over $AP$ rather
than explicit symbols in $2^{AP}$, with binary decision diagrams providing an
efficient representation~\citep{akers1978binary}.


\begin{example}
\label{ch02:examp:NBA}
Consider the specification
$\varphi = (\square\Diamond a_1) \wedge (\square\Diamond a_2)
\wedge (\square\Diamond a_3) \wedge (\square \neg a_4)$.
A corresponding automaton encodes repeated satisfaction of the surveillance
propositions together with permanent exclusion of $a_4$. If an edge from
$q_1$ to $q_3$ is labeled by $a_2 \wedge \neg a_4$, then every symbol in
$2^{AP}$ satisfying that expression enables the transition; one symbolic label
therefore represents several concrete alphabet symbols.
\end{example}

\subsection{Weighted product automaton}
\label{ch02:subsec:product}

The transition system $\mathcal{T}_c$ captures feasible motion, while
$\mathcal{A}_{\varphi}$ captures the task language. Their product yields a
single model whose accepting runs encode satisfying robot behaviors.

\begin{definition}
\label{ch02:def:product}
The weighted product B\"uchi automaton of $\mathcal{T}_c$ and
$\mathcal{A}_{\varphi}$ is denoted by
$\mathcal{A}_{p} \triangleq \mathcal{T}_c \otimes \mathcal{A}_{\varphi}$ and
defined by:
\begin{equation}
\mathcal{A}_{p}
\triangleq
\left(Q',\,\delta',\,Q_0',\,\mathcal{F}',\,w_p\right),
\label{ch02:eq:product}
\end{equation}
where
$Q' \triangleq \Pi \times Q$,
$Q_0' \triangleq \Pi_0 \times Q_0$,
$\mathcal{F}' \triangleq \Pi \times \mathcal{F}$, and
$\delta' : Q' \to 2^{Q'}$ satisfies
\begin{equation}
\langle \pi_j,q_n \rangle \in \delta'(\langle \pi_i,q_m \rangle)
\iff
\pi_i \longrightarrow_c \pi_j
\text{ and }
q_n \in \delta\!\left(q_m,L_c(\pi_i)\right),
\label{ch02:eq:product_transition}
\end{equation}
where the transition weight is defined by:
\begin{equation}
w_p\bigl(\langle \pi_i,q_m \rangle,\langle \pi_j,q_n \rangle\bigr)
\triangleq
w_c(\pi_i,\pi_j),
\label{ch02:eq:product_weight}
\end{equation}
for every enabled transition in \eqref{ch02:eq:product_transition}.
\end{definition}

Equation~\eqref{ch02:eq:product_transition} enables a product transition only
when the robot can move from $\pi_i$ to $\pi_j$ and the task automaton can move
from $q_m$ to $q_n$ under the label at $\pi_i$. The weight assignment in
\eqref{ch02:eq:product_weight} preserves the motion cost from $\mathcal{T}_c$.
Thus satisfaction and optimality can be analyzed directly on the product.

\begin{definition}
\label{ch02:def:projection}
Let $q' = \langle \pi,q \rangle \in Q'$. The projection of $q'$ onto the
transition-system component is defined by
$q'|_{\Pi} \triangleq \pi$, and the projection onto the automaton component is
defined by
$q'|_{Q} \triangleq q$.
Given an infinite run
$R = q_0'q_1'q_2'\cdots$ of $\mathcal{A}_p$, its projected sequences are
denoted by
$R|_{\Pi} \triangleq q_0'|_{\Pi}q_1'|_{\Pi}q_2'|_{\Pi}\cdots$
and
$R|_{Q} \triangleq q_0'|_{Q}q_1'|_{Q}q_2'|_{Q}\cdots$.
\end{definition}

The product construction converts synthesis into graph search over
$\mathcal{A}_p$: an accepting product run projects to a plan of
$\mathcal{T}_c$ satisfying the task formula.
Conversely, any path of $\mathcal{T}_c$ whose trace satisfies $\varphi$ induces
accepting nonemptiness of the product. Synthesis therefore reduces to computing
an accepting run with the required structure.

\begin{lemma}
\label{ch02:lem:satisfy}
If $R$ is an accepting run of $\mathcal{A}_p$, then
$R|_{\Pi} \models \varphi$.
\end{lemma}

\begin{proof}
Since $R$ is accepting, some state in $\mathcal{F}'$ appears infinitely often
in $R$. By Definition~\ref{ch02:def:projection}, the projection of that state
onto $Q$ belongs to $\mathcal{F}$. Hence $R|_{Q}$ is an accepting run of
$\mathcal{A}_{\varphi}$. By \eqref{ch02:eq:product_transition}, the run
$R|_{Q}$ is generated by the word
$\texttt{trace}(R|_{\Pi})$. Therefore,
$\texttt{trace}(R|_{\Pi})
\in \mathcal{L}_{\omega}(\mathcal{A}_{\varphi})$.
Applying Lemma~\ref{ch02:lem:LTL-BA} yields
$\texttt{trace}(R|_{\Pi}) \models \varphi$, which implies
$R|_{\Pi} \models \varphi$ by
Definition~\ref{ch02:def:path-satisfy}.
\end{proof}


\subsection{Optimal accepting run search}
\label{ch02:subsec:optimality}

Problem~\ref{ch02:prob:nominal} requires prefix-suffix structure. Since the
product automaton is finite, accepting behaviors admit ultimately periodic
representations, so the search may be restricted to lasso-shaped runs.

\begin{lemma}
\label{ch02:lem:prefix-suffix}
If there exists an infinite path $\tau$ of $\mathcal{T}_c$ with prefix-suffix
structure such that $\tau \models \varphi$, then there exists an accepting run
of $\mathcal{A}_p$ with prefix-suffix structure.
\end{lemma}

\begin{proof}
Since $\tau \models \varphi$, the word
$\sigma \triangleq \texttt{trace}(\tau)$ belongs to
$\texttt{Words}(\varphi)$. By Lemma~\ref{ch02:lem:LTL-BA},
$\sigma \in \mathcal{L}_{\omega}(\mathcal{A}_{\varphi})$. Therefore,
$\sigma$ induces an accepting run of $\mathcal{A}_{\varphi}$. Pairing the path
$\tau$ with such a resulting automaton run through
\eqref{ch02:eq:product_transition} yields a run of $\mathcal{A}_p$. Since both
components are ultimately periodic, the product run is also ultimately
periodic. Because the run is accepting, some accepting product state is visited
infinitely often and can be selected as the root of the suffix cycle.
\end{proof}

Accordingly, the search is restricted to accepting runs of the form below:
\begin{equation}
R \triangleq
q_0' q_1' \cdots q_{f-1}'
\left[q_f' q_{f+1}' \cdots q_n'\right]^{\omega},
\label{ch02:eq:prod-Apath}
\end{equation}
where $q_0' \in Q_0'$,
$q_f' \in \mathcal{F}'$, and the transition from $q_n'$ to $q_f'$ is enabled.
The representation in \eqref{ch02:eq:prod-Apath} gives a finite description of
an infinite accepting run: the prefix reaches $q_f'$ once, and the suffix rooted
at $q_f'$ repeats indefinitely.

\begin{definition}
\label{ch02:def:run_cost}
Let $R$ be an accepting run of the form
\eqref{ch02:eq:prod-Apath}. Its total cost is defined by:
\begin{equation}
J(R)
\triangleq
\sum_{i=0}^{f-1} w_p(q_i',q_{i+1}')
+
\gamma
\left(
\sum_{i=f}^{n-1} w_p(q_i',q_{i+1}')
+
w_p(q_n',q_f')
\right),
\label{ch02:eq:prod-cost}
\end{equation}
where $\gamma \geq 0$ is a design parameter that specifies the relative weight
of the suffix cost with respect to the prefix cost.
\end{definition}

The first term in \eqref{ch02:eq:prod-cost} is the transient cost of reaching
an accepting state, while the second is the cost of one recurrent suffix cycle.
Larger $\gamma$ therefore emphasizes long-term performance.

\begin{problem}
\label{ch02:pro:opt-search}
Given the product automaton $\mathcal{A}_p$, determine an accepting run of the
form \eqref{ch02:eq:prod-Apath} that minimizes the cost in
\eqref{ch02:eq:prod-cost}.
\end{problem}

The search decomposes into two finite graph problems: a shortest prefix path
from an initial product state to an accepting state, and a shortest suffix cycle
rooted at that accepting state. Since the weights are nonnegative,
Dijkstra-type search applies. The product may be constructed fully, by
generating all states and transitions in advance, or on the fly, by generating
adjacency only when a state is visited. The latter is useful when much of the
product graph is irrelevant to the optimal accepting run.
Let $S' \subseteq Q_0'$ denote the set of initial product states considered
by the search. For each starting state $q_s' \in S'$ and each accepting state
$q_f' \in \mathcal{F}'$, let
$d_{\textup{pre}}(q_s',q_f')$
denote the cost of a shortest path from $q_s'$ to $q_f'$ in $\mathcal{A}_p$.
Let
$d_{\textup{suf}}(q_f')$
denote the cost of a shortest cycle rooted at $q_f'$. Let
$R_{\textup{pre}}(q_s',q_f')$
and
$R_{\textup{suf}}(q_f')$
denote the corresponding prefix path and suffix cycle, respectively. Then the
cost of the accepting run induced by the pair $(q_s',q_f')$ is given by:
\begin{equation}
J(q_s',q_f')
\triangleq
d_{\textup{pre}}(q_s',q_f')
+
\gamma\, d_{\textup{suf}}(q_f'),
\label{ch02:eq:run_pair_cost}
\end{equation}
where $\gamma \geq 0$ is the weighting parameter introduced in
\eqref{ch02:eq:prod-cost}. The search problem is therefore reduced to finding a
pair $(q_s',q_f')$ that minimizes \eqref{ch02:eq:run_pair_cost}.

%==============================
%==============================
\begin{algorithm}[t]
\caption{Optimal accepting run search, \texttt{OptRun}().}
\label{ch02:alg:optimal}
\DontPrintSemicolon
\KwIn{$\mathcal{A}_p$, $S' \subseteq Q_0'$, $\gamma \geq 0$}
\KwOut{$R_{\textup{opt}}$ or failure}

\If{$S'=\emptyset$ or $\mathcal{F}'=\emptyset$}{
  \Return{failure}\;
}

$J^\star \leftarrow +\infty$\;
$R_{\textup{opt}} \leftarrow \emptyset$\;

\ForEach{$q_f' \in \mathcal{F}'$}{
  $(d_{\textup{suf}}(q_f'),\, R_{\textup{suf}}(q_f'))
  \leftarrow
  \texttt{DijksCycle}(\mathcal{A}_p, q_f')$\;
}

\ForEach{$q_s' \in S'$}{
  $(d_{\textup{pre}}(q_s',\cdot),\, R_{\textup{pre}}(q_s',\cdot))
  \leftarrow
  \texttt{DijksTargets}(\mathcal{A}_p, q_s', \mathcal{F}')$\;
  \ForEach{$q_f' \in \mathcal{F}'$}{
    \If{$d_{\textup{{pre}}}(q_s',q_f') < +\infty$
    \textup{ and }
    $d_{\textup{{suf}}}(q_f') < +\infty$}{
      $J \leftarrow d_{\textup{pre}}(q_s',q_f')
      + \gamma\, d_{\textup{suf}}(q_f')$\;
      \If{$J < J^\star$}{
        $J^\star \leftarrow J$\;
        $R_{\textup{opt}} \leftarrow
        R_{\textup{pre}}(q_s',q_f')
        \bigl[R_{\textup{suf}}(q_f')\bigr]^{\omega}$\;
      }
    }
  }
}

\If{$R_{\textup{opt}}=\emptyset$}{
  \Return{failure}\;
}
\Return{$R_{\textup{opt}}$}\;
\end{algorithm}
%==============================
%==============================

Algorithm~\ref{ch02:alg:optimal} first computes the shortest suffix cycle for
each accepting state, then the shortest prefix paths from each
$q_s' \in S'$ to reachable accepting states. Each feasible pair
$(q_s',q_f')$ defines a candidate lasso with cost
\eqref{ch02:eq:run_pair_cost}; the minimizing pair gives
$R_{\textup{opt}}$, whose symbolic plan is
$\tau_{\textup{opt}} \triangleq R_{\textup{opt}}|_{\Pi}$.


\begin{remark}
\label{ch02:rem:optimality_loss}
The projected plan $\tau_{\textup{opt}}$ is optimal with respect to the search
performed on $\mathcal{A}_p$, but it need not coincide with the globally
optimal satisfying plan in $\mathcal{T}_c$. This possible loss of optimality is
caused by the translation from LTL to a nondeterministic B\"uchi automaton.
This phenomenon is discussed in~\citep{schuppan2005shortest}. Tighter automaton
constructions may preserve optimality more faithfully, but such constructions
typically produce larger automata and hence higher computational cost.
\end{remark}


\subsection{Hybrid execution of the optimal run}
\label{ch02:sec:hybrid}

The output of Algorithm~\ref{ch02:alg:optimal} is a symbolic accepting run that
must be realized in the continuous workspace by the controller in
Definition~\ref{ch02:def:transition}. The hybrid controller activates the
appropriate local controller for each consecutive region pair in the projected
plan. Let the optimal accepting run be written as
\begin{equation}
R_{\textup{opt}}
\triangleq
R_{\textup{pre}}
\left[R_{\textup{suf}}\right]^{\omega},
\label{ch02:eq:final-path}
\end{equation}
where $R_{\textup{pre}}$ is the finite prefix and $R_{\textup{suf}}$ is the
finite suffix cycle.
The projected motion plan is
$\tau_{\textup{opt}} \triangleq R_{\textup{opt}}|_{\Pi}$. Execution follows the
prefix once, then switches to the suffix and repeats it indefinitely, realizing
the structure required in \eqref{ch02:eq:prefix-suffix}.

For implementation, a segment variable and an index locate the current position
within $R_{\textup{opt}}$. The helper function $\texttt{NextGoal}(\cdot)$
returns the next product state; projection onto $\Pi$ gives the next region;
and \eqref{ch02:eq:control} generates the active controller.
Algorithm~\ref{ch02:alg:hybrid} runs indefinitely because the suffix repeats
infinitely often. Once $x(t) \in \pi_{\textup{next}}$ is confirmed by the
localization mechanism, the controller advances to the next symbolic goal.

%==============================
%==============================
\begin{algorithm}[t]
\caption{Plan execution by hybrid control, \texttt{HybCon}().}
\label{ch02:alg:hybrid}
\DontPrintSemicolon
\KwIn{$R_{\textup{opt}}$, current state measurement $x(t)$}
\KwOut{$u(t)$, $R_{\textup{past}}$, $\tau_{\textup{past}}$}
initialize
$q_{\textup{cur}}' = q_{\textup{next}}' = R_{\textup{pre},1}$,
$\pi_{\textup{cur}} = q_{\textup{cur}}'|_{\Pi}$,
$\pi_{\textup{next}} = q_{\textup{next}}'|_{\Pi}$,
$R_{\textup{past}} = [\;]$,
$\tau_{\textup{past}} = [\;]$\;
\While{\textup{\texttt{True}}}{
  \If{$x(t) \in \pi_{\textup{next}}$ is confirmed}{
    $q_{\textup{cur}}' = q_{\textup{next}}'$\;
    append $q_{\textup{cur}}'$ to $R_{\textup{past}}$\;
    append $\pi_{\textup{next}}$ to $\tau_{\textup{past}}$\;
    $q_{\textup{next}}' =
    \texttt{NextGoal}(q_{\textup{cur}}',R_{\textup{opt}})$\;
    $\pi_{\textup{cur}} = q_{\textup{cur}}'|_{\Pi}$\;
    $\pi_{\textup{next}} = q_{\textup{next}}'|_{\Pi}$\;
  }
  $u(t) =
  \mathcal{U}\bigl(x(t),\pi_{\textup{cur}},\pi_{\textup{next}}\bigr)$\;
}
\end{algorithm}
%==============================
%==============================


\subsection{Complexity discussion}
\label{ch02:sssec:complexity_on_the_fly}

The computational burden is determined mainly by the task automaton and product
automaton sizes. Let $n_{\varphi} \triangleq |\varphi|$ denote the size of the
LTL formula. Standard translation bounds imply:
$|Q| = 2^{\mathcal{O}(n_{\varphi})}$ and $|\delta| = 2^{\mathcal{O}(n_{\varphi})}$,
where $Q$ is the automaton state set and $|\delta|$ denotes the number of
enabled automaton transitions in $\mathcal{A}_{\varphi}$ from
Definition~\ref{ch02:def:NBA}.
Once $\mathcal{T}_c$ and $\mathcal{A}_{\varphi}$ are fixed, the fully
constructed product automaton satisfies:
$|Q'| = |\Pi|\,|Q|$ and $|\delta'| = \mathcal{O}\bigl(|\longrightarrow_c|\,|\delta|\bigr)$,
where $|\Pi|$ and $|\longrightarrow_c|$ denote the numbers of states and
transitions in $\mathcal{T}_c$.

Under on-the-fly construction, only the portion of the product graph that is
actually visited during search is generated. Let
$Q'_{\textup{vis}} \subseteq Q'$
denote the set of visited product states, and let
$\delta'_{\textup{vis}} \subseteq \delta'$
denote the set of product transitions that are explicitly constructed. Then the
effective synthesis time is bounded by:
$
T_{\textup{otf}}
=
\mathcal{O}
\Bigl(
\bigl(|Q_0'| + |\mathcal{F}'_{\textup{vis}}|\bigr)
\,|\delta'_{\textup{vis}}|\,
\log |Q'_{\textup{vis}}|
\Bigr)$,
where
$\mathcal{F}'_{\textup{vis}} \triangleq \mathcal{F}' \cap Q'_{\textup{vis}}$
is the set of accepting product states that are actually generated.
The corresponding memory requirement is
$
M_{\textup{otf}}
=\mathcal{O}\bigl(|Q'_{\textup{vis}}| + |\delta'_{\textup{vis}}|\bigr)$,
for the explicitly stored product states and transitions. In the worst case,
on-the-fly construction has the same asymptotic complexity as full
construction, since the search may visit the entire product graph. In practice,
it is often more efficient because many product states are unreachable or
irrelevant, especially in adaptive settings where the transition system is
updated repeatedly.

\begin{example}
\label{ch02:exam:complexity}
Consider grid-based transition systems with $n^2$ states and adjacent-region
costs drawn uniformly from $[0,1]$. The task transports two objects to two
destinations and then returns to a base region, following the same general form
as the later case study. For each instance, $t_{\textup{full}}$ records full
product construction time, $t_{\textup{syn}}$ the optimal-run search time on
the full product, and $t_{\textup{fly}}$ the on-the-fly search time. The
associated B\"uchi automaton has $75$ states and $877$ transitions.
Figure~\ref{fig:ch02:single_robot_complexity} shows that product construction
dominates the computation, motivating symbolic and on-the-fly constructions.
\end{example}

\subsection{Scalability in the multi-robot setting}
\label{ch02:sssec:multi_robot_scalability}

For a single robot, the synthesis procedure is tractable when the workspace
abstraction and task automaton are moderate in size. A centralized multi-robot
extension is substantially less scalable.
Consider $N$ robots with local transition systems
$\mathcal{T}_i \triangleq
\left(\Pi_i,\longrightarrow_i,\Pi_{i,0},AP_i,L_i,w_i\right)$,
where $i \in \{1,\cdots,N\}$. A centralized team abstraction yields the joint
state space
$\Pi_{\textup{team}} \triangleq \prod_{i=1}^{N} \Pi_i$.
Hence,
$|\Pi_{\textup{team}}| = \prod_{i=1}^{N} |\Pi_i|$,
which grows multiplicatively with the number of robots.
If a team specification $\Phi$ is translated into a B\"uchi automaton
$\mathcal{A}_{\Phi}$ with state set $Q_{\Phi}$, then the corresponding
centralized product space has cardinality below:
\begin{equation}
|Q_{\textup{team}}'|
=
|Q_{\Phi}|
\prod_{i=1}^{N} |\Pi_i|\approx 2^{|\Phi|} M^N,
\label{ch02:eq:team-product-cardinality}
\end{equation}
where the multiplicative growth arises from both the joint motion state and the
task automaton;
and $M$ is the size of homogeneous $\mathcal{T}_i$.
Equation~\eqref{ch02:eq:team-product-cardinality} shows why direct centralized
synthesis quickly becomes prohibitive: even small local models can yield a team
product too large for explicit construction and exhaustive search. This
motivates the decentralized and compositional developments in later chapters,
and is consistent with scalability limits reported in multi-robot
temporal-logic planning
\citep{kloetzer2010automatic,chen2012formal,kloetzer2011multi,guo2015multi}.
Example~\ref{ch02:exam:multi_robot_complexity} gives a numerical illustration.

\begin{example}
\label{ch02:exam:multi_robot_complexity}
Consider $N$ identical robots, each modeled by a $5 \times 5$ grid transition
system with $|\Pi_i| = 25$, and let the team task use the $75$-state,
$877$-transition B\"uchi automaton from Example~\ref{ch02:exam:complexity}.
Then $|\Pi_{\textup{team}}| = 25^N$ and
$|Q'_{\textup{team}}| = 75 \cdot 25^N$, so the state space grows exponentially
even though the local model and task automaton remain fixed,
as shown in Figure~\ref{fig:ch02:multi_robot_complexity}.
Linear extrapolation from the single-robot benchmark shows rapid impracticality: with
six robots, the estimated product-construction time is about $58$ days and the
estimated synthesis time exceeds $2.7$ days. Direct centralized synthesis is
therefore unsuitable beyond small teams.
\end{example}

\section{Case Study}
\label{ch02:sec:case_study}

This case study instantiates the nominal synthesis pipeline on a concrete
single-robot problem, showing how workspace abstraction, temporal-task
specification, product construction, and hybrid execution combine in one
baseline instance for the later multi-robot developments.

%==============================
\subsection{Workspace and platform}
\label{ch02:subsec:case_workspace}

The workspace is a structured indoor environment of size
$2.4\,\mathrm{m} \times 2.1\,\mathrm{m}$ with two rows of three rooms separated
by a corridor partitioned into three segments. The region set is
$\Pi \triangleq \{r_1,\cdots,r_6,c_1,c_2,c_3\}$, where
$r_1,r_2,r_3$ lie on one side, $r_4,r_5,r_6$ on the other, and
$c_1,c_2,c_3$ represent the corridor. This partition preserves the adjacency
relations needed for motion planning. The associated atomic propositions are
$\textsf{r1},\cdots,\textsf{r6},\textsf{c1},\textsf{c2},\textsf{c3}$, together
with the object propositions $\textsf{rball}$, $\textsf{gball}$, and
$\textsf{basket}$. The labeling map is specified by
$L_c(r_1)=\{\textsf{r1}\}$,
$L_c(r_2)=\{\textsf{r2},\textsf{basket}\}$,
$L_c(r_3)=\{\textsf{r3},\textsf{gball}\}$,
$L_c(c_1)=\{\textsf{c1}\}$,
$L_c(c_2)=\{\textsf{c2}\}$,
$L_c(c_3)=\{\textsf{c3}\}$,
$L_c(r_4)=\{\textsf{r4},\textsf{basket}\}$,
$L_c(r_5)=\{\textsf{r5},\textsf{rball}\}$, and
$L_c(r_6)=\{\textsf{r6}\}$.
The cost of a transition between adjacent regions is taken as the Euclidean
distance between their centers.

The platform is TIAGo. At the abstraction level, execution is represented by
$\mathcal{T}_c$; low-level realization is handled by the platform-specific
motion and action interface in Section~\ref{ch02:subsec:tiago_execution}. The
generic controller $\mathcal{U}(x(t),\pi_i,\pi_j)$ in \eqref{ch02:eq:control}
is instantiated so adjacent-region transitions are executable by the navigation
and action stack. The resulting $\mathcal{T}_c$ has $9$ states and
$16$ transitions; simulation adds Gaussian actuation perturbations to model
execution noise.

\subsection{Platform-level execution on TIAGo}
\label{ch02:subsec:tiago_execution}

The repository \texttt{tiago\_ltl\_flexbe}~\citep{guo_tiago_ltl_flexbe}
illustrates the nominal framework on the TIAGo platform. It shows that temporal
logic plans need not remain abstract region sequences: they can be grounded in
a platform-specific stack that combines navigation, supervisory behavior
coordination, and predefined actions. Temporal tasks may therefore include both
motion objectives and actions such as grasping, gesturing, or inspection at
designated locations. Figure~\ref{ch02:fig:tiago_execution} shows a
representative execution scene in Gazebo and RViz.
An illustrative task example from the TIAGo implementation is
\begin{equation}
\phi_{\textup{ex}}
\triangleq
\Diamond\Bigl(
(\textsf{r2} \wedge \textsf{pick\_from\_floor})
\wedge
\Diamond(\textsf{r3} \wedge \textsf{reach\_max})
\Bigr),
\label{ch02:eq:tiago_example}
\end{equation}
where the robot first reaches $r_2$ and executes
\textsf{pick\_from\_floor}, then reaches $r_3$ and executes
\textsf{reach\_max}. At the symbolic level, a corresponding plan has the form
$\pi_{\textup{init}}
\rightarrow
(r_2,\textsf{pick\_from\_floor})
\rightarrow
(r_3,\textsf{reach\_max})$,
which shows how region goals and actions are combined in the platform-level
execution stack; see \citep{guo_tiago_ltl_flexbe} for implementation details.

\section{Summary}
\label{ch02:sec:ch02_summary}


This chapter established the nominal single-robot framework used
throughout the rest of the monograph. Robot motion was abstracted as a
control-driven weighted transition system, tasks were specified in LTL,
and the resulting synthesis problem was solved through B\"uchi-automaton
construction, product search, and hybrid execution. The scalability
discussion also highlighted why these foundations must later be extended
through decentralized, compositional, and reactive coordination
mechanisms in the multi-robot setting.
