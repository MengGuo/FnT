\section{Case Study}
\label{sec:ch04_case_study}

This section illustrates the top-down coordination framework of this
chapter through two representative studies. The first is a large-scale
continual mission with heterogeneous robots, intended to show how
hierarchical decomposition, abstract team assignment, team formation,
and local coordination interact under persistent online demand. The
second is a hardware validation in an industrial emergency-response
scenario, used to verify that the same framework remains implementable
under real sensing, communication, and execution constraints.

%==============================
\subsection{Large-scale continual mission with heterogeneous robots}
\label{ch04:subsec:case_large_scale}

This simulation study focuses on the hierarchical coordination framework
of Section~\ref{ch04:subsec:task}. In particular, it illustrates the
uncertain mission model of Section~\ref{ch04:subsec:uncertain_mission},
the simultaneous decomposition and assignment method of
Section~\ref{ch04:subsubsec:simultaneous_assignment}, the
capacity-based team-formation procedure of
Section~\ref{ch04:subsubsec:team_formation}, and the local
coordination strategies of Section~\ref{ch04:subsec:local_coordination}.
The mission contains a sustained stream of heterogeneous demands, so
newly released tasks must be absorbed while previously assigned tasks
are still under execution.

%==============================
\subsubsection{Workspace and system description}
\label{ch04:subsubsec:case_large_scale_workspace}

The simulation is conducted in a two-dimensional environment of size
$130\,\mathrm{m} \times 40\,\mathrm{m}$. The fleet contains
$80$ robots of three heterogeneous types, denoted by \texttt{uavA},
\texttt{uavB}, and \texttt{uavC}. These types differ in their
primitive action sets. In the considered instance, the relevant actions
are perception, delivery, and grasping, with
$A_{\mathrm{uavA}} = \{\textsf{perception},\textsf{delivery}\},
A_{\mathrm{uavB}} = \{\textsf{perception},\textsf{grasping}\}$,
$A_{\mathrm{uavC}} = \{\textsf{delivery},\textsf{grasping}\}$,
where the three robot types are not interchangeable in either the
global assignment stage or the local coordination stage. Surveillance
tasks require perception-capable robots, whereas delivery and
capture-related tasks require delivery or grasping capabilities,
possibly from several robots acting collaboratively.

The overall workload contains $30$ collaborative tasks, further
refined into $450$ subtasks, so that the average decomposition ratio
is $15$ subtasks per collaborative task. Each collaborative task
$\omega_m$ follows the model introduced in
Section~\ref{ch04:subsec:uncertain_mission}.
The mission contains three
classes of collaborative tasks. The first class consists of static and
known delivery tasks, for which the required subtasks and their
locations are available before execution. The second class consists of
static and unknown surveillance tasks, for which the execution region
is known but the actual targets or inspection points are revealed only
during execution. The third class consists of dynamic and known
capture tasks, for which the target identities are known initially but
their states evolve over time. These classes correspond directly to the
local coordination cases developed in
Section~\ref{ch04:subsec:local_coordination}.
The temporal specification of each released mission is expressed over
collaborative task propositions. A representative mission template is
given by:
\[ \varphi_r =
\Diamond\!\left(
\varphi^r_{\mathrm{del}}
\wedge
\Diamond \varphi^r_{\mathrm{surv}}
\right)
\wedge
\left(
\neg \varphi^r_{\mathrm{cap}}\,
\mathsf{U}\,
\varphi^r_{\mathrm{surv}}
\right),
\]
where $\varphi^r_{\mathrm{del}}$ encodes delivery requirements,
$\varphi^r_{\mathrm{surv}}$ encodes surveillance requirements, and
$\varphi^r_{\mathrm{cap}}$ encodes capture-related requirements.
Each of these formulas is grounded by one or more collaborative tasks
$\omega_m$, and each task further induces concrete subtask
requirements of the form $(n_{mj},a_{mj},s_{mj})$. Thus, the
temporal formula specifies the ordering and logical dependencies among
task classes, whereas the collaborative task model specifies the
capability, cardinality, and spatial requirements that must be
satisfied by the assigned team.

To emulate a persistent mission, the tasks are not released
simultaneously. Instead, six release waves occur at
$0.0\,\mathrm{s}$, $30.0\,\mathrm{s}$, $55.0\,\mathrm{s}$,
$80.0\,\mathrm{s}$, $105.0\,\mathrm{s}$, and
$130.0\,\mathrm{s}$. Around each release instant, the active mission
set is updated, the planner revises the task decomposition, and the
subteam structure is recomputed for the unstarted and newly released
tasks. Tasks that are already under execution are not preempted. This
non-preemptive rule preserves execution continuity while still
allowing the fleet to react to newly released tasks and changing
resource requirements.
Figure~\ref{ch04:fig:case_large_scale_overview} provides an overview
of the mission instance. The labeled nodes
$\omega_1,\cdots,\omega_{30}$ denote the collaborative tasks, and the
different shapes indicate the different task classes. The grouped
nodes on the right summarize the higher-level structures produced by
hierarchical coordination. The dense set of gray links illustrates the
coupling and inheritance relations induced by temporal precedence,
shared capabilities, and resource-flow dependencies. These couplings
make a direct flat assignment unattractive at this scale, since each
robot-to-subtask decision would have to account simultaneously for
temporal feasibility, team capacity, motion cost, and online task
release.

\subsubsection{Performance and replanning efficiency}
\label{ch04:subsubsec:case_large_scale_performance}

The simulation completes all $30$ collaborative tasks successfully,
yielding a task success rate of $100\%$. The final makespan is
$154.4\,\mathrm{s}$, and the reported average task response time is
$24.65\,\mathrm{s}$. Measured at the workload level, the effective
throughput is approximately $0.194$ collaborative tasks per second.
Measured at the subtask level, the execution rate is approximately
$2.91$ subtasks per second. These figures indicate that the proposed
hierarchical scheme remains operational even when fleet size, task
heterogeneity, and online release are all substantial.
A more detailed view is obtained by grouping the response times by
release wave. For the six task waves, the average response times are
$24.7\,\mathrm{s}$, $24.2\,\mathrm{s}$, $22.6\,\mathrm{s}$,
$18.9\,\mathrm{s}$, $33.1\,\mathrm{s}$, and $24.4\,\mathrm{s}$,
respectively, with a wave-averaged value of $24.65\,\mathrm{s}$. The
largest value occurs at the release wave around $105.0\,\mathrm{s}$,
where the response time rises to $33.1\,\mathrm{s}$. Relative to the
wave-averaged value, this is an increase of about $34.3\%$, which
indicates that this stage is the most congested portion of the mission.
Even so, the system remains feasible and eventually absorbs the arriving
workload without leaving unresolved tasks at the end of execution.

Online replanning is triggered $8$ times in total. The corresponding
replanning times are
$0.752\,\mathrm{s}$, $0.677\,\mathrm{s}$, $0.688\,\mathrm{s}$,
$0.749\,\mathrm{s}$, $0.053\,\mathrm{s}$, $0.159\,\mathrm{s}$,
$0.204\,\mathrm{s}$, and $0.273\,\mathrm{s}$. Hence the total
replanning time is $3.555\,\mathrm{s}$, the average replanning time is
$0.444\,\mathrm{s}$, and the maximum single replanning time is
$0.752\,\mathrm{s}$. Relative to the overall makespan, the cumulative
replanning overhead is only about $2.30\%$. When normalized by workload
size, the average replanning cost is $0.12\,\mathrm{s}$ per
collaborative task, or about $0.008\,\mathrm{s}$ per subtask. These
values show that the online adaptation layer does not dominate the total
execution time, which is consistent with the complexity control built
into the decomposition-and-assignment procedure of
Section~\ref{ch04:subsubsec:simultaneous_assignment}.
Moreover, Figure~\ref{ch04:fig:case_large_scale_gantt} shows the task Gantt chart.
The four colors represent release, navigation, synchronization, and
execution, respectively. The red polyline reports the number of active
subteams over time. Its peak value is approximately $6$, which indicates
that the planner activates only a limited number of concurrent execution
units even though the fleet itself is much larger. This behavior is
consistent with the capacity-aware team-formation policy of
Section~\ref{ch04:subsubsec:team_formation}: many candidate teams
may be generated, but only a bounded subset is activated in parallel at
a given time.

\subsubsection{Execution structure and resource reorganization}
\label{ch04:subsubsec:case_large_scale_structure}

The aggregate system states provide additional evidence of stable
execution. The average occupancy levels of the principal execution
states are $8.64$ for travelling, $3.98$ for ready, and $25.33$ for
working. In addition, all $80$ robots remain available throughout the
simulation. Thus, no large-scale resource loss or unrecoverable team
collapse occurs during the mission. The dominant time consumption lies
in motion and coordinated execution rather than in replanning, which is
again consistent with the design goal of separating expensive
mission-level reasoning from local execution updates.
Figure~\ref{ch04:fig:case_large_scale_states} reports the stacked state
statistics. Each major workload injection produces a characteristic
cycle: newly arrived tasks first appear as unassigned workload, then
migrate into the assigned set after replanning, and finally decrease as
the corresponding teams complete execution. This recurring
arrival--replan--assign--execute pattern shows that the system does not
accumulate a growing unresolved backlog over time.
Instead, the system prevents the released workload from accumulating into a
persistent backlog, even when some waves overlap temporally with later releases.
The most prominent disturbance again appears near $105.0\,\mathrm{s}$,
where the rise in unassigned and assigned workload coincides with the
largest wave-level response time. The red dashed lines in the figure
mark all $8$ replanning instants, including both the $6$ task-release
updates and the additional threshold-triggered replanning events.

A complementary view is provided by the Sankey diagram in
Fig.~\ref{ch04:fig:case_large_scale_sankey}. The left side represents
individual robots, the middle layers represent intermediate subteams
formed during planning and regrouping, and the right side represents the
high-level execution units that carry the mission through its later
stages. The varying edge widths indicate different amounts of member
flow and capability transfer. This figure makes explicit a central claim
of Section~\ref{ch04:subsec:task}: hierarchical coordination is not merely
a static partition of the fleet, but a dynamic reorganization process in
which subteams may be created, inherited, merged, or reconfigured as the
mission evolves. Compared with a flat individual-level scheduler, such a
layered structure is substantially better suited to large fleets under
mixed static-known, static-unknown, and dynamic-known workloads.

Overall, these results confirm the practical role of the methods
developed in Section~\ref{ch04:subsec:task}. The uncertain mission model
supports persistent online arrivals, the simultaneous decomposition and
assignment stage controls the combinatorial growth of the planning
problem, the team-formation layer enables bounded parallel execution,
and the three local coordination modes absorb task uncertainty at the
team level. In combination, these mechanisms yield a scalable and
computationally efficient coordination architecture for large-scale
heterogeneous fleets under sustained mission demand.

\subsection{Hardware validation in an industrial emergency-response
scenario}
\label{ch04:subsec:hardware_validation}

To complement the simulation study, a hardware validation was conducted
in a realistic industrial emergency-response scenario. The purpose is to
verify that the hierarchical coordination framework developed in
Section~\ref{ch04:subsec:task} can be implemented on a heterogeneous
physical system under real sensing, communication, and execution
constraints. In particular, the deployment instantiates the uncertain
mission model of Section~\ref{ch04:subsec:uncertain_mission}, the
simultaneous decomposition-and-assignment mechanism of
Section~\ref{ch04:subsubsec:simultaneous_assignment}, the
team-formation logic of
Section~\ref{ch04:subsubsec:team_formation}, and the local
coordination procedures of Section~\ref{ch04:subsec:local_coordination}.

%==============================
%==============================
\subsubsection{Workspace and system description}
The experimental site measures $16\,\mathrm{m} \times 18\,\mathrm{m}$ and
emulates an industrial plant with pipelines, storage tanks, fire events,
injured personnel, and chemical leaks, as shown in
Figure~\ref{ch04:fig:hardware_validation_architecture}.
The validation instance uses $8$
robots, including $2$ UAVs, $4$ UGVs, and $2$ legged robots, supervised
by $3$ tablet-equipped operators. The task set contains four event
classes: fire suppression, casualty handling, chemical containment, and
equipment protection. Representative mission fragments are
\begin{align*}
\varphi_{\mathrm{af}}
&=\Diamond(\mathrm{af}\wedge
\Diamond(\mathrm{insp}_{\mathrm{af}}\wedge
\Diamond((\mathrm{cut}_{\mathrm{valve}}\vee
\mathrm{cool}_{\mathrm{water}})\wedge
\Diamond\mathrm{monitor}_{\mathrm{af}}))),\\
\varphi_{\mathrm{tp}}
&=\Diamond(\mathrm{tp}\wedge
\Diamond(\mathrm{insp}_{\mathrm{tp}}\wedge
\Diamond(\mathrm{clean\_up}_{\mathrm{tp}}\wedge
\Diamond(\mathrm{rescue}_{\mathrm{tp}}\wedge
\Diamond\mathrm{monitor}_{\mathrm{tp}})))),\\
\varphi_{\mathrm{tank}}
&=\Diamond(\mathrm{tank}\wedge
\Diamond(\mathrm{cool}_{\mathrm{tank}}\wedge
\Diamond(\mathrm{fix}_{\mathrm{tank}}\wedge
\Diamond\mathrm{monitor}_{\mathrm{tank}}))),
\end{align*}
where $\Diamond$ denotes eventual satisfaction; $\mathrm{af}$,
$\mathrm{tp}$, and $\mathrm{tank}$ denote an alkane flame, a trapped
person, and a damaged tank; and the subscripts specify the event to which
each inspection, intervention, rescue, repair, or monitoring action
belongs. The full mission specification conjoins these fragments with the
remaining fire, leakage, and casualty formulas and is converted into an
R-poset for scheduling and monitoring.
The induced priority and concurrency relations are summarized as
$\{\omega_{\mathrm{tp}},\omega_{\mathrm{poi}}\}
\prec \Omega_{\mathrm{hw}}\setminus
\{\omega_{\mathrm{tp}},\omega_{\mathrm{poi}}\},
\omega_{\mathrm{af}}\prec\omega_{\mathrm{htlf}},
\omega_{\mathrm{hvf}}\parallel\omega_{\mathrm{h2s}}\parallel
\omega_{\mathrm{tank}}$,
where $\Omega_{\mathrm{hw}}$ is the active hardware-mission set,
$\omega_{\mathrm{poi}}$ denotes poisoned-person rescue,
$\omega_{\mathrm{htlf}}$ denotes high-temperature liquid-fire response,
$\omega_{\mathrm{hvf}}$ denotes high-voltage fire response, and
$\prec$ and $\parallel$ denote precedence and admissible parallelism.
Thus life-saving missions have highest priority, alkane-flame mitigation
precedes high-temperature liquid-fire handling, and high-voltage fire,
hydrogen-sulfide leakage, and tank protection may proceed concurrently
when resources do not conflict. The corresponding collaboration
requirements are encoded as skill-count constraints, e.g.,
$
\mathcal{C}_{\mathrm{af}}
=\{(\mathrm{inspect},2),(\mathrm{operate},1),
(\mathrm{liquid\_spray},1),(\mathrm{monitor},2)\},
\mathcal{C}_{\mathrm{tp}}
=\{(\mathrm{inspect},2),(\mathrm{clean\_up},2),
(\mathrm{rescue},1),(\mathrm{monitor},2)\},
\mathcal{C}_{\mathrm{h2s}}
=\{(\mathrm{inspect},2),(\mathrm{ignite},1),
(\mathrm{solid\_spray},1)$,
$(\mathrm{monitor},2)\},\mathcal{C}_{\mathrm{tank}}
=\{(\mathrm{liquid\_spray},1),(\mathrm{monitor},2),
(\mathrm{fix},1)\}$,
where each pair gives a required skill and the number of robots that must
provide it. A subtask is released only when the selected group satisfies
these counts and has the required resource. The cloud layer maintains the
mission DAG, priorities, and shared resource state; the edge layer solves
local subtask allocation under precedence and skill-count constraints; and
the robot layer executes platform-specific primitives. In implementation,
cloud and edge programs run on a workstation operated through tablets,
while ROS1 UAVs/UGVs and ROS2 legged robots communicate through dedicated
TCP/UDP bridges and execute their own perception, localization, and motion
planning stacks.

\subsubsection{Experimental results}
The hardware trial lasts approximately $11$ minutes and contains $8$
tasks with $33$ subtasks in total. As shown in Figure~\ref{ch04:fig:hardware_validation_results},
during execution, the system identifies
major emergency events, including fires, injured personnel, and
damaged tanks and leakage cues. The cloud layer is invoked $6$ times for mission
updates and cross-group coordination, whereas the edge layer is triggered
$44$ times for local task processing. The average edge-side assignment
time is $0.5\,\mathrm{s}$, the average subtask execution time is
$20\,\mathrm{s}$, and the average task completion time is
$82.5\,\mathrm{s}$. These values are consistent with the intended
separation of time scales: cloud-level reasoning handles relatively
infrequent mission updates and cross-group resource sharing, edge-level
reasoning performs repeated local interpretation and scheduling, and the
robot layer executes physical subtasks in real time. Two representative
episodes highlight the role of hierarchical reorganization. In the first,
an injured person is detected during fire suppression; the event is
interpreted as a higher-priority rescue mission, escalated, and reassigned
to a more suitable group. In the second, a legged robot is dispatched to
fix a damaged storage tank through a dedicated platform-level action
primitive. These episodes verify that the formal task hierarchy is not
only synthesized offline, but also updated online when new task instances,
new resources, or platform-specific execution requirements appear. Across
$10$ hardware trials, the framework achieves a task success rate of
$100\%$. The average end-to-end planning time is $21.6\,\mathrm{s}$,
and the plan consistency is $95\%$. Each trial requires about $50$ operator
interactions on average. Relative to fully manual operation, the reported
operator workload decreases by $56\%$, from $7.5$ to $3.3$. These results
indicate that the framework developed in this chapter transfers to the
hardware setting without noticeable degradation in planning quality,
while substantially reducing operator burden in a realistic multi-robot
emergency-response environment.

\section{Summary}
\label{ch04:sec:summary}

This chapter developed a top-down framework for global temporal-task
coordination in multi-robot systems. Section~\ref{ch04:sec:po_decomp_assignment}
introduced automaton pruning, task decomposition, relaxed partially ordered
sets, and branch-and-bound assignment for a fixed global mission.
Section~\ref{ch04:subsec:task} extended this framework to large heterogeneous
fleets through hierarchical mission decomposition, abstract team assignment,
team formation, and local coordination within teams.
Section~\ref{ch04:sec:online_continual_tasks} then addressed continual tasks by
updating the task structure online through R-poset products and by applying a
receding-horizon replanning scheme. Human requests were incorporated as special
online task updates within the same framework.
